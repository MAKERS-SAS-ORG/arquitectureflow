# ADR-001: Multi-LLM con abstraccion de proveedor (Claude + GPT-4)

## Roles colaboradores

> :busts_in_silhouette: **Especialista Tecnico (Tech Lead)** principal — aporto consecuencias
> tecnicas y validacion del DPA. :busts_in_silhouette: **Compliance** valido las clausulas de
> retencion de datos.

## Estado
Aceptado

## Contexto y Problema

El RFC-001 selecciono "plataforma propia con LLM via API". Necesitamos decidir el
proveedor LLM y la estrategia de vendor lock-in. La regulacion exige que datos
sensibles de salud no queden retenidos por terceros, y el negocio quiere evitar
quedarse atado a un solo proveedor.

## Decision Drivers
- **DPA con zero-retention:** ambos proveedores deben firmar acuerdo de no entrenamiento.
- **Latencia p95 < 2.5s** para mantener experiencia conversacional fluida.
- **Calidad en espanol latinoamericano:** las pruebas internas mostraron Claude > GPT-4 en variantes regionales (validar formal en POC).
- **Resiliencia:** si un proveedor cae, el sistema debe degradar — no caerse.
- **Costo:** $35M COP/mes target (~50k sesiones, promedio 8 turnos).

## Opciones Consideradas
- Opcion 1: Solo Claude (Anthropic) via Bedrock
- Opcion 2: Solo GPT-4 (OpenAI) via Azure OpenAI
- Opcion 3: Multi-LLM con abstraccion (Claude primario, GPT-4 fallback)

## Decision

**Opcion elegida: "Multi-LLM con abstraccion (Opcion 3)"**

### Justificacion
Porque (1) ningun proveedor solo cumple las garantias de disponibilidad necesarias
para un canal regulado; (2) la abstraccion permite cambiar proveedor primario sin
re-arquitectar; y (3) Compliance prefiere evitar dependencia unica. A diferencia
de las opciones mono-vendor, esta agrega complejidad pero habilita rollover en
minutos si un proveedor tiene incidente.

## Pros y Contras

### Opcion 1: Solo Claude via Bedrock
- (+) Mejor calidad en espanol regional (POC interno).
- (+) DPA zero-retention de Anthropic (verificar firma — :red_circle: TODO).
- (-) Lock-in con AWS Bedrock.
- (-) Sin fallback si AWS Bedrock cae.

### Opcion 2: Solo GPT-4 via Azure OpenAI
- (+) Tiers de capacidad reservados.
- (+) DPA Microsoft mas maduro para sector regulado.
- (-) Calidad espanol regional inferior en POC.
- (-) Costo por token mas alto ~22%.

### Opcion 3: Multi-LLM con abstraccion (elegida)
- (+) Resiliencia (failover entre proveedores).
- (+) Capacidad de A/B test entre proveedores con prompt versioning.
- (+) Negociacion comercial mas fuerte.
- (-) Complejidad arquitectonica (provider abstraction layer).
- (-) Requiere mantener prompts equivalentes en 2 proveedores.

## Consecuencias

### Positivas
- Resiliencia operacional — el canal no depende de un proveedor.
- Habilita migracion futura a modelo open-source (Llama 3, Mixtral) en Fase 2.
- Auditoria trazable por proveedor.

### Negativas / Trade-offs
- **Mas codigo:** Provider abstraction layer (PAL) que el equipo debe mantener.
- **Mas pruebas:** cada cambio de prompt debe probarse en ambos proveedores.
- **Costo cognitivo:** el equipo debe entender ambas APIs (Bedrock + Azure OpenAI).

### Neutrales
- Observabilidad debe distinguir metricas por proveedor (queries, latencia, errores).

## Validacion

> Como sabremos en 6 meses que esta decision fue correcta?

- **Metrica:** % de sesiones servidas correctamente cuando un proveedor falla (target: > 99%).
- **Senal de alerta:** si menos del 5% del trafico va al proveedor secundario en 90 dias,
  la abstraccion no esta justificando su costo — re-evaluar.
- **Revision programada:** 2026-12-15 (1 mes post-go-live).
