# PRD: AsistIA — Asistente Conversacional de Reclamos

## Roles colaboradores

> :busts_in_silhouette: **Acelerador** (casos de uso) + :busts_in_silhouette: **QA** (escenarios Gherkin
> + Quality Attribute Scenarios) + :busts_in_silhouette: **Compliance** (regla de consentimiento).
> Primer artefacto donde QA aparece — su trabajo aqui evita reescribir despues.

## 1. Objetivo del Producto
> Permitir que el asegurado radique un reclamo de salud en menos de 5 minutos a
> traves de una conversacion guiada con IA, manteniendo audit trail y cumplimiento
> regulatorio.

## 2. Usuarios Objetivo

| Persona | Descripcion | Necesidad Principal |
|---|---|---|
| Asegurado titular | 25-65 anos, mobile-first, baja paciencia para formularios | Radicar rapido sin llamar al call center |
| Autorizado familiar | Cuida a beneficiario (hijo/padre), tiene poder | Radicar en nombre del titular con validacion |
| Evaluador interno (SaludVita) | Profesional de salud / auditor | Ver traza completa para decidir casos ambiguos |

## 3. Funcionalidades (Casos de Uso)

### CU-001: Radicacion guiada de reclamo ambulatorio
- **Actores:** Asegurado, AsistIA (agente LLM), Core de Polizas, Evaluador (si aplica)
- **Precondicion:** asegurado autenticado + consentimiento Habeas Data aceptado
- **Flujo principal:**
  1. AsistIA saluda y solicita consentimiento de tratamiento (CONS-001 — Compliance)
  2. AsistIA pide tipo de servicio (consulta, examen, urgencia)
  3. AsistIA solicita prestador, fecha, monto y soportes
  4. AsistIA consulta cobertura al Core de Polizas
  5. Si claro: AsistIA confirma y genera RIPS automaticamente (auto-radicacion)
  6. Si ambiguo o negado: AsistIA notifica que sera revisado por evaluador en < 2h
- **Flujos alternativos:**
  - Alt-A: usuario sube soportes ilegibles → AsistIA pide nuevamente
  - Alt-B: Core de Polizas no responde → AsistIA registra reclamo offline y notifica
- **Postcondicion:** reclamo en estado "Radicado" o "En revision evaluador"
- **Prioridad:** Must Have

### CU-002: Carga de soportes medicos
- **Actores:** Asegurado, AsistIA, OCR, S3
- **Flujo principal:** asegurado fotografia/sube PDF → OCR extrae monto/fecha/prestador → AsistIA confirma extraccion
- **Prioridad:** Must Have

### CU-003: Consulta de estado de reclamo previo
- **Actores:** Asegurado, AsistIA, CRM
- **Prioridad:** Should Have

## 4. User Stories con Criterios de Aceptacion

### US-001: Consentimiento Habeas Data antes de cualquier dato sensible
**Como** asegurado, **quiero** dar consentimiento explicito antes de que el sistema
procese mis datos de salud, **para** estar tranquilo de que mis derechos estan protegidos.

**Criterios de Aceptacion** (validados con :busts_in_silhouette: **Compliance** y :busts_in_silhouette: **QA**):

```gherkin
Scenario: Consentimiento aceptado antes de iniciar
  Given el asegurado abre AsistIA por primera vez
  When el sistema saluda
  Then debe presentar el texto de consentimiento Ley 1581/2012 ANTES de cualquier
       pregunta sobre salud
  And debe registrar fecha, hora, IP, agente, version del texto en audit trail

Scenario: Consentimiento rechazado
  Given el asegurado abre AsistIA
  When rechaza el consentimiento
  Then el sistema cierra la sesion con mensaje cordial
  And NO registra ningun dato adicional
  And ofrece canal alternativo (call center)

Scenario: Consentimiento revocado durante sesion
  Given el asegurado tiene una sesion en curso
  When pide explicitamente "elimina mis datos"
  Then el sistema borra los datos de la sesion en curso (Right to be forgotten)
  And persiste solo el audit trail de revocacion (regulatorio)
```

### US-002: Auto-radicacion de casos claros
**Como** asegurado, **quiero** que reclamos simples se radiquen al instante,
**para** no esperar a un evaluador humano cuando todo esta claro.

```gherkin
Scenario: Servicio cubierto, prestador en red, sin alertas
  Given asegurado con poliza activa que cubre consultas
  And reclamo a prestador en red por consulta ambulatoria
  When AsistIA consulta el Core
  And el Core retorna "Cubierto al 100%"
  Then AsistIA radica automaticamente (estado: Radicado)
  And genera RIPS valido
  And notifica al asegurado en < 2 segundos despues de la confirmacion

Scenario: Caso ambiguo
  Given asegurado con servicio que requiere autorizacion
  When AsistIA consulta el Core
  And el Core retorna "Requiere revision medica"
  Then AsistIA NO radica
  And enruta a evaluador
  And informa al asegurado: SLA < 2h habiles
```

## 5. Quality Attribute Scenarios (NFRs)

> Aterrizado con :busts_in_silhouette: **QA** y :busts_in_silhouette: **Compliance**. Formato Bass et al.

### QA-001: Performance — Latencia conversacional
| Campo | Valor |
|---|---|
| Fuente | Asegurado en sesion activa |
| Estimulo | Envia un mensaje |
| Artefacto | Pipeline AsistIA → LLM → Core → respuesta |
| Entorno | Carga normal (1.000 sesiones concurrentes) |
| Respuesta | Token streaming inicia |
| Medida | Time-to-first-token (TTFT) < 1.5s p95 / Latencia total < 3s p95 |

### QA-002: Reliability — Disponibilidad
| Campo | Valor |
|---|---|
| Fuente | Cualquier asegurado |
| Estimulo | Inicia sesion en cualquier momento |
| Artefacto | Sistema completo |
| Entorno | 24/7 |
| Respuesta | Sesion exitosa O fallback a canal legacy si falla |
| Medida | Disponibilidad >= 99.5% mensual; **0% de sesiones bloqueadas sin alternativa** |

### QA-003: Security — Confidencialidad de datos sensibles
| Campo | Valor |
|---|---|
| Fuente | Atacante externo |
| Estimulo | Intenta acceder a datos de otra sesion / extraer datos via prompt injection |
| Artefacto | API + LLM provider |
| Entorno | Produccion |
| Respuesta | Bloquea + alerta + audit |
| Medida | 0 incidentes de filtracion confirmados; 100% de datos PII enmascarados antes de enviar al LLM |

### QA-004: Compliance — Audit trail inmutable
| Campo | Valor |
|---|---|
| Fuente | SuperSalud / SIC en auditoria |
| Estimulo | Solicita historial de un reclamo |
| Artefacto | Audit log |
| Entorno | Hasta 10 anos despues |
| Respuesta | Recupera traza completa con timestamps |
| Medida | Audit trail 100% completo + WORM (Write Once Read Many) confirmado |

## 6. Reglas de Negocio

- RN-001: Consentimiento Habeas Data MUST capturarse ANTES de cualquier pregunta sobre salud.
- RN-002: AsistIA NO PUEDE decidir negar cobertura — solo evaluador humano (ADR-002).
- RN-003: Datos sensibles (diagnosticos, medicamentos) MUST anonimizarse antes de enviar al LLM (PII redaction).
- RN-004: Toda sesion conversacional MUST persistirse con WORM por 10 anos (SuperSalud).
- RN-005: Si una sesion supera 30 minutos, el sistema sugiere transferir a un agente humano.

## 7. Fuera de Scope (explicito)
- Reclamos de **odontologia/vision** — Fase 2.
- **Voz / IVR** — solo chat texto en MVP.
- **Pago automatico** del reclamo — sigue manual por Operaciones (decision SuperSalud).
- **Apelaciones** de reclamos negados — flujo separado.
- **Lengua indigena** — no MVP.

## 8. Dependencias y Restricciones
- Depende de: Core de Polizas SOAP (legacy), API de Pagos, RIPS endpoint, S3 para soportes.
- Restricciones: Habeas Data, Resolucion SuperSalud 2266/2024, NIST AI RMF.
