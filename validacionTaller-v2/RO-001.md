# Requisitos Operacionales: AsistIA

> El SA define QUE operar; DevOps define COMO.

## Roles colaboradores

> :busts_in_silhouette: **DevOps / SRE Lead** principal — health check viable, rollback automatizable,
> integracion con plataforma de monitoreo. Este artefacto se construyo en una sesion conjunta.

## 1. Informacion del Sistema

| Item | Valor |
|---|---|
| URL produccion | `https://asistia.saludvita.co` (pendiente DNS) |
| URL staging | `https://asistia-stg.saludvita.co` |
| Region/Zona | `sa-east-1` primaria, `us-east-1` DR |
| Dashboard de monitoreo | CloudWatch + Datadog (links pendientes) |
| Contacto de escalada | Oncall SaludVita Plataforma — PagerDuty `team-asistia` |

## 2. Requisitos de Deployment

| Requisito | Valor |
|---|---|
| Estrategia | Blue-green via CodeDeploy con feature flag (LaunchDarkly) |
| Tiempo maximo de deployment | 15 minutos por servicio |
| Health check endpoint | `/healthz` — retorna `200 {"status":"ok","deps":[...]}` |
| Health check criterio | DB conectada + Bedrock alcanzable + Core SOAP alcanzable |

### Checklist post-despliegue

- [ ] Health check `/healthz` retorna OK por 5 min
- [ ] Tasa de error < 0.5%
- [ ] Flujo principal funciona: radicar reclamo de prueba E2E (smoke test scripted)
- [ ] Metricas de tokens LLM no exceden 1.5x baseline

## 3. Criterios de Rollback

| Condicion | Accion |
|---|---|
| Tasa de error > 2% por 3 minutos | Rollback automatico via feature flag |
| Health check falla 60s post-deploy | Rollback automatico |
| TTFT chat > 4s p95 sostenido 5 min | Evaluacion manual, posible rollback |
| % consentimientos no firmados > 0 | **Rollback inmediato — violacion regulatoria** |

**Ventana de rollback segura:** 6 horas post-deploy (mas alla, las migraciones de schema requieren plan especial).

## 4. Metricas Criticas y Umbrales

| Metrica | Umbral de alarma | Severidad | Escalar a |
|---|---|---|---|
| Disponibilidad `chat-api` | < 99.5% (5min) | P1 | Oncall + CTO |
| TTFT > 2.5s p95 | sostenido 5 min | P2 | Oncall |
| Failover LLM > 5% | sostenido 1h | P2 | Oncall + Tech Lead |
| Audit writer DLQ > 0 | cualquier mensaje | **P1** | Oncall + Compliance |
| PII redaction violation | cualquier evento | **P1** | Oncall + Security + Compliance |
| Costo tokens > $1.5M COP/dia | sostenido 24h | P3 | Tech Lead |

### Niveles de severidad
| Nivel | Definicion | Tiempo de respuesta |
|---|---|---|
| P1 | Sistema caido / violacion regulatoria | < 15 minutos |
| P2 | Funcionalidad degradada | < 1 hora |
| P3 | Issue menor | Proximo dia habil |

## 5. Dependencias Externas y SLAs

| Sistema | SLA contratado | Timeout | Comportamiento si falla |
|---|---|---|---|
| Core de Polizas (interno) | 99% | 3s | Reclamo en estado "En revision evaluador" |
| Bedrock (Claude) | 99.9% | 5s | Failover a Azure OpenAI |
| Azure OpenAI (GPT-4) | 99.9% | 5s | Mensaje cordial + canal alternativo |
| Sistema de Pagos | 99.9% | 2s | Queue retry async |
| RIPS SuperSalud | Best-effort | -- | Retry + alerta Compliance |

## 6. Politica de Gestion de Cambios

Todo cambio en produccion MUST:
- Tener ticket aprobado con descripcion del cambio y plan de rollback.
- Ejecutarse en ventana de bajo trafico: **22:00-05:00 hora Colombia**.
- Tener feature flag asociado para rollback inmediato.

Cambios que requieren aprobacion adicional del SA:
- Cambios en system prompt (SPS-001) — requieren tambien firma de Compliance.
- Cambios en regla de "claridad" para auto-radicacion (ADR-002).
- Cambios en politica de PII redaction (RN-003).
- Cambios en contratos con sistemas externos.
- Migraciones de schema en `audit-store`.
