# Requisitos Operacionales: Plataforma de Inversion en Renta Fija

> Este documento define los REQUISITOS que Operations/SRE necesita para
> crear el runbook detallado. El SA define QUE operar; Operations define COMO.

## 1. Informacion del Sistema

| Item | Valor |
|---|---|
| URL produccion | https://app.inversiones.example.com |
| URL staging | https://staging.inversiones.example.com |
| Region | AWS us-east-1 |
| Dashboard de monitoreo | CloudWatch Dashboard "inversiones-prod" |
| Contacto de escalada | Tech Lead — canal #inversiones-ops en Slack |

## 2. Requisitos de Deployment

| Requisito | Valor |
|---|---|
| Estrategia | Rolling update (ECS) |
| Tiempo maximo de deployment | 15 minutos |
| Health check endpoint | GET /health |
| Health check criterio | HTTP 200, respuesta < 500ms |

### Checklist de verificacion post-despliegue
- [ ] GET /health retorna 200
- [ ] Tasa de error < 0.1% en los ultimos 5 minutos
- [ ] Crear inversion de prueba en staging y verificar confirmacion
- [ ] Audit trail registra la operacion de prueba
- [ ] SQS queue depth estable (no creciendo)

## 3. Criterios de Rollback

| Condicion | Accion |
|---|---|
| Tasa de error > 1% por mas de 3 minutos | Rollback automatico (ECS rolling) |
| Health check falla en 60 segundos post-deploy | Rollback automatico |
| Latencia P99 > 2s por mas de 5 minutos | Evaluacion manual, posible rollback |
| SQS DLQ recibe > 10 mensajes en 10 minutos | Investigar, posible rollback |

**Ventana de rollback segura:** 30 minutos post-deploy (ECS mantiene imagen anterior)

## 4. Metricas Criticas y Umbrales

| Metrica | Umbral de alarma | Severidad | Escalar a |
|---|---|---|---|
| Disponibilidad API | < 99.5% en 1 hora | P1 | Tech Lead + On-call |
| Latencia P99 | > 1s por 5 min | P2 | On-call |
| Tasa de error API | > 1% por 5 min | P1 | Tech Lead + On-call |
| SQS queue depth | > 200 por 5 min | P2 | On-call |
| Deceval error rate | > 20% por 10 min | P2 | On-call + Tech Lead |
| RDS CPU | > 80% por 5 min | P2 | On-call |
| DLQ message count | > 0 | P3 | On-call (proximo dia habil) |

### Niveles de severidad
| Nivel | Definicion | Tiempo de respuesta |
|---|---|---|
| P1 | Sistema caido, datos financieros en riesgo, o SLO roto | < 15 minutos |
| P2 | Funcionalidad degradada, operaciones lentas | < 1 hora |
| P3 | Issue menor, mensajes en DLQ, metricas anomalas | Proximo dia habil |

## 5. Dependencias Externas y SLAs

| Sistema | SLA contratado | Timeout | Comportamiento si falla |
|---|---|---|---|
| Deceval | 99% | 15 seg | Reintento auto en T+5min, usuario ve "En Proceso" |
| PSE/ACH | 99.5% | 30 seg | Reintento auto x2, luego notificar usuario |
| Proveedor KYC | 99% | 10 seg | Cola de verificacion manual |
| SFC reportes | Sin SLA real-time | 60 seg | Batch diario, no impacta operacion |

## 6. Politica de Gestion de Cambios

Todo cambio en produccion MUST:
- Tener ticket aprobado con descripcion del cambio
- Tener plan de rollback documentado
- Ejecutarse en ventana de bajo trafico: lunes a jueves, 6:00-8:00 AM COT

Cambios que requieren aprobacion adicional del SA:
- Cambios en schema de base de datos
- Cambios en contratos de API (endpoints, payloads)
- Cambios en integraciones con Deceval, PSE o KYC
- Actualizaciones de .NET o PostgreSQL (major version)
- Cualquier cambio que afecte el audit trail
