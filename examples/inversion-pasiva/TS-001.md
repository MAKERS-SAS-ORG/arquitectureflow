# Tech Spec: Plataforma de Inversion en Renta Fija

## 1. Stack Tecnologico

| Capa | Tecnologia | Version | Justificacion |
|---|---|---|---|
| API Backend | .NET | 8.0 LTS | Equipo tiene experiencia, ecosistema financiero maduro |
| Base de datos | PostgreSQL | 16 | Open source, soporte transaccional fuerte, JSON native |
| Cola de mensajes | Amazon SQS | -- | Managed, escalabilidad automatica (ADR-001) |
| Cache | Redis (ElastiCache) | 7.x | Sesiones, rate limiting, cache de catalogo |
| Frontend | React/Next.js | 14.x | SPA con SSR para SEO de onboarding |
| Infraestructura | AWS (ECS Fargate) | -- | Serverless containers, sin gestion de servidores |

## 2. Estructura de Contenedores (C4 Level 2)

> **Diagrama C4 L2 generado con Excalidraw MCP** — ver seccion de diagramas abajo.

### Contenedores

| Contenedor | Tecnologia | Responsabilidad | Tipo |
|---|---|---|---|
| Web App | Next.js | UI para inversionistas: onboarding, catalogo, portafolio | Frontend |
| API Gateway | AWS ALB | Ruteo, TLS termination, rate limiting | Infra |
| Investment API | .NET 8 | Endpoints REST para operaciones de inversion | API |
| Worker de Operaciones | .NET 8 | Procesa operaciones async: KYC, pagos, custodia | Worker |
| PostgreSQL | RDS PostgreSQL 16 | Datos de usuarios, inversiones, transacciones, audit trail | Database |
| Redis | ElastiCache | Cache de catalogo de CDTs, sesiones, rate limiting | Cache |
| Cola SQS | Amazon SQS | Mensajes de operaciones pendientes + DLQ | Queue |

## 3. Contratos de Integracion

### Contrato: Web App -> Investment API
| Campo | Valor |
|---|---|
| Protocolo | REST/HTTPS |
| Autenticacion | JWT (expiracion 15 min) |
| Endpoints principales | POST /inversiones, GET /inversiones/{id}, GET /portafolio |
| Respuesta a creacion | 202 Accepted {id, estado: "Recibida"} |

### Contrato: Investment API -> Cola SQS
| Campo | Valor |
|---|---|
| Protocolo | AWS SDK (async) |
| Formato | JSON |
| Idempotencia | Idempotency-Key en header (UUID v4, generado por cliente) |
| DLQ | Configurada, reintentos: 3, backoff: exponencial |

### Contrato: Worker -> Proveedor de Custodia (Deceval)
| Campo | Valor |
|---|---|
| Proveedor | Deceval |
| Protocolo | REST/HTTPS |
| SLA contratado | 99% disponibilidad, latencia < 10s |
| Timeout | 15 segundos |
| Fallback | Enqueue para reintento en T+5min, notificar al usuario |

### Contrato: Worker -> Pasarela de Pagos (PSE)
| Campo | Valor |
|---|---|
| Proveedor | ACH Colombia (PSE) |
| Protocolo | REST/HTTPS |
| SLA contratado | 99.5% |
| Timeout | 30 segundos |
| Fallback | Reintento automatico x2, luego notificar al usuario |

## 4. Quality Attribute Trade-offs

| Trade-off | Decision | Justificacion |
|---|---|---|
| Consistencia vs UX | Eventual consistency + feedback inmediato | El usuario ve "En Proceso" -> "Confirmada". Mejor UX que esperar 30s. |
| Simplicidad vs Resiliencia | Event-driven con SQS | Mayor complejidad operacional pero resiliente a fallos de proveedores (ADR-001) |
| Costo vs Escalabilidad | ECS Fargate (pay-per-use) | Sin servidores que administrar, escala a 0 en baja demanda |

## 5. Fitness Functions

| FF | Atributo | Metrica | Umbral | Automatizacion |
|---|---|---|---|---|
| FF-001 | Performance | Latencia API p99 | < 500ms | Load test en CI (k6) |
| FF-002 | Reliability | SQS queue depth | < 200 mensajes | CloudWatch alarm |
| FF-003 | Security | Vulnerabilidades criticas | 0 | SAST en pipeline (Snyk) |
| FF-004 | Compliance | Audit trail completeness | 100% operaciones | Test de integracion |

## 6. Convenciones a Nivel de Solucion

- **APIs:** RESTful, versionadas via URL (/v1/), formato de error RFC 7807
- **Logging:** JSON estructurado, correlation-id en todos los logs
- **Secretos:** AWS Secrets Manager (produccion), .NET User Secrets (local)
- **Idempotencia:** Todas las escrituras aceptan Idempotency-Key header
