# ADR-001: Arquitectura event-driven con API REST para plataforma de inversiones

> **Origen:** CB-001 (restriccion: equipo .NET + AWS), RFC-001 seccion 5.1 (sistemas externos)

## Estado
Aceptado

## Contexto y Problema

La plataforma de inversion en renta fija (RFC-001, opcion C aprobada) necesita procesar
operaciones financieras que involucran multiples sistemas externos identificados en
CB-001: Deceval (custodia), PSE/ACH (pagos), proveedor KYC y SFC (regulador).

Las operaciones de inversion NO son instantaneas: comprar un CDT requiere verificar KYC,
debitar fondos, registrar en custodia y confirmar al usuario. Este flujo puede tomar
segundos a minutos dependiendo de los proveedores externos.

Necesitamos decidir el patron de comunicacion principal del sistema.

## Decision Drivers
- Las operaciones financieras MUST ser trazables y auditables
- Los proveedores externos tienen latencias variables (1s - 30s)
- El sistema MUST poder reintentar operaciones fallidas sin duplicar transacciones
- El usuario SHOULD recibir feedback inmediato aunque el proceso sea asincrono
- MUST soportar 500 operaciones/hora en pico

## Opciones Consideradas
- Opcion 1: API sincrona end-to-end (request-response)
- Opcion 2: Event-driven con cola de mensajes + API REST para frontend
- Opcion 3: Full event sourcing con CQRS

## Decision

**Opcion elegida: "Event-driven con cola de mensajes + API REST para frontend"**

### Justificacion
Porque la naturaleza asincrona de las operaciones financieras se alinea con
el patron event-driven (los proveedores externos no responden instantaneamente),
y una API REST para el frontend permite feedback inmediato (202 Accepted)
con polling de estado.

A diferencia de la Opcion 1 (sincrona) que bloquearia al usuario mientras espera
respuesta de custodia (hasta 30s), y de la Opcion 3 (full event sourcing) que
agrega complejidad innecesaria para el volumen actual (500 ops/hora).

## Consecuencias

### Positivas
- Desacoplamiento temporal entre frontend y procesos backend
- Reintentos naturales via dead-letter queue
- Audit trail implicito en los eventos
- El usuario recibe confirmacion inmediata (202) sin esperar proveedores

### Negativas / Trade-offs
- Consistencia eventual: el usuario ve "En Proceso" antes de "Confirmada"
- Complejidad adicional: cola de mensajes, workers, manejo de DLQ
- Debugging mas dificil: flujo distribuido vs request-response simple

### Neutrales
- Requiere estrategia de idempotencia (ver futuro ADR-002)
- Requiere monitoreo de profundidad de cola y latencia de procesamiento

## Validacion
- **Metrica:** Tiempo promedio de procesamiento de inversion < 2 minutos E2E
- **Senal de alerta:** Cola con profundidad > 200 mensajes por mas de 5 minutos
- **Revision programada:** 6 meses post-lanzamiento (evaluar si el volumen justifica event sourcing)
