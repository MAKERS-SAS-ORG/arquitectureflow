# ADR-002: PostgreSQL como Base de Datos Principal

## Estado

Aceptado

## Contexto y Problema

La plataforma de micro-inversion en CDTs necesita una base de datos relacional para
almacenar: usuarios, portafolios, transacciones financieras y audit trail inmutable.
Las opciones viables del equipo son PostgreSQL y SQL Server, dado que el equipo trabaja
en .NET 8 (CB-001) y ambos motores son compatibles con Entity Framework Core 8.

La base de datos debe soportar:
- Row-Level Security nativo (aislamiento de datos por usuario — RN-002 del PRD-001)
- Tabla append-only para audit trail inmutable (RN-003 del PRD-001)
- JSONB para esquemas semi-estructurados (eventos de dominio del CM-001)
- Retencion de datos de 10 anos (regulacion financiera colombiana — CB-001 seccion 4)

El presupuesto total del proyecto es $1.200M COP (CB-001 seccion 3), lo que hace
que el costo de infraestructura sea un factor diferenciador critico entre las opciones.

## Decision Drivers

- **Costo de infraestructura:** Presupuesto limitado a $1.200M COP (CB-001) — las licencias
  de base de datos impactan directamente el runway del proyecto
- **Row-Level Security nativo:** RN-002 del PRD-001 exige aislamiento de datos a nivel de
  base de datos, no solo a nivel de aplicacion
- **Experiencia del equipo:** 5 ingenieros .NET con experiencia mixta en ambos motores
  (mas experiencia historica en SQL Server, pero conocimiento basico de PostgreSQL)
- **Compatibilidad AWS RDS:** Ambos disponibles en RDS, pero costos y modelos de
  licenciamiento son significativamente diferentes
- **Soporte para datos semi-estructurados:** Eventos de dominio del CM-001 requieren
  almacenamiento flexible sin migraciones de schema por cada tipo de evento

## Opciones Consideradas

- Opcion 1: SQL Server en RDS
- Opcion 2: PostgreSQL en RDS

## Decision

**Opcion elegida: "PostgreSQL en RDS"**

### Justificacion

PostgreSQL ofrece Row-Level Security nativo (critico para FF-004), JSONB para eventos
de dominio sin schema rigido, y costo 60% menor que SQL Server en RDS (license-included).
Entity Framework Core 8 soporta PostgreSQL via Npgsql con paridad completa de features,
incluyendo migraciones, LINQ queries y connection pooling. El ahorro de ~$3.240 USD/ano
en infraestructura es reinvertible en el proyecto dado el presupuesto limitado.

## Pros y Contras de las Opciones

### Opcion 1: SQL Server en RDS

- (+) El equipo lo conoce mejor historicamente (menor curva de aprendizaje inicial)
- (+) Integracion nativa con .NET sin paquete NuGet adicional
- (+) SSMS como herramienta de debug con la que el equipo ya es productivo
- (-) Costo 2.5x mayor en RDS license-included (~$450 USD/mes vs ~$180 USD/mes para db.r6g.large)
- (-) Row-Level Security disponible pero con configuracion mas compleja y menos documentacion
- (-) Licenciamiento adicional si se escala a replicas de lectura (costo compuesto)
- (-) Vendor lock-in con Microsoft para scaling futuro

### Opcion 2: PostgreSQL en RDS -- ELEGIDA

- (+) Row-Level Security nativo, maduro y bien documentado
- (+) JSONB para almacenar eventos de dominio sin migraciones de schema
- (+) 60% mas barato en RDS ($180 vs $450 USD/mes en instancia equivalente db.r6g.large)
- (+) Open source — sin riesgo de licenciamiento al escalar replicas o instancias
- (+) Entity Framework Core 8 + Npgsql es production-ready y activamente mantenido
- (+) Ecosistema de extensiones (pg_cron para jobs de retencion, pgcrypto para cifrado)
- (-) Equipo necesita 1-2 semanas de familiarizacion con particularidades de PostgreSQL
- (-) Tooling de debug ligeramente diferente (pgAdmin vs SSMS — curva de aprendizaje menor)

## Consecuencias

### Positivas

- Ahorro de ~$3.240 USD/ano en infraestructura (reinvertible en desarrollo del producto)
- RLS nativo simplifica la implementacion de FF-004 (aislamiento de datos por usuario)
- JSONB permite almacenar domain events sin migraciones de schema por cada tipo nuevo
- Sin riesgo de licenciamiento al escalar horizontalmente con replicas de lectura

### Negativas / Trade-offs

- Sprint 0 incluira 2-3 dias de setup y familiarizacion del equipo con PostgreSQL + Npgsql
- Operaciones debe aprender pgAdmin y particularidades de pg_dump/pg_restore para backups
- Algunas queries T-SQL especificas no son portables directamente (CTEs recursivos, funciones de ventana con sintaxis diferente)

### Neutrales

- Multi-AZ y backups automaticos funcionan igual en RDS para ambos motores
- Performance para el volumen esperado (300 inversiones/dia, 50K usuarios) es equivalente en ambos
- Tanto SQL Server como PostgreSQL soportan cifrado en reposo via RDS encryption (Ley 1581)

## Validacion

- **Metrica:** Tiempo de queries criticas (portafolio, transacciones, audit trail) < 50ms p99
- **Senal de alerta:** Queries que requieran workarounds significativos por diferencias PostgreSQL vs SQL Server, o caida de velocidad del equipo por la curva de aprendizaje
- **Revision programada:** Sprint 2 — evaluar si la familiarizacion impacto la velocidad del equipo y si las queries criticas cumplen el umbral de 50ms
