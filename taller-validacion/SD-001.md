## 2. Drivers de Arquitectura

### 2.1 Requisitos de Rendimiento

| Escenario | Latencia objetivo | Throughput | Medicion |
|---|---|---|---|
| Consultar portafolio | < 200ms P99 | 500 req/seg pico | P99 en CloudWatch + X-Ray traces |
| Compra de CDT (API) | < 500ms P99 (202 Accepted) | 100 req/seg pico | P99 en CloudWatch + X-Ray traces |
| Proceso E2E de inversion | < 5 min P95 | 120 inversiones/min | CloudWatch custom metric con timestamps de saga |
| Onboarding + KYC | < 45s verificacion | 300/hora pico | P95 en CloudWatch |
| Listado de CDTs disponibles | < 150ms P99 | 800 req/seg pico | Cache hit rate + P99 en CloudWatch |

### 2.2 Disponibilidad y Tolerancia a Fallos

| Componente | Disponibilidad | Estrategia |
|---|---|---|
| Investment API (ECS Fargate) | 99.5% | Multi-AZ (3 AZs), health checks c/10s, auto-scaling por CPU y request count |
| PostgreSQL (RDS) | 99.95% | Multi-AZ con failover automatico, automated backups c/hora, read replicas |
| Redis (ElastiCache) | 99.9% | Cluster mode enabled, Multi-AZ, automatic failover |
| SQS/SNS | 99.99% | Managed AWS, sin accion requerida |
| ALB | 99.99% | Managed AWS, Multi-AZ |

**RPO:** 1 hora (backups continuos de RDS + WAL archiving a S3)
**RTO:** 30 minutos (failover automatico Multi-AZ RDS + redeploy ECS con imagen cached en ECR)

**Error Budget:** Con SLO de 99.5% para la API, el error budget mensual es de 3.6 horas/mes.
Cuando el error budget se consume al 80%, se congela el deployment de features y se prioriza estabilidad.

### 2.3 Escalabilidad

| Horizonte | Usuarios | Inversiones/hora | Estrategia |
|---|---|---|---|
| Lanzamiento | 1.000 | 50 | 2 ECS tasks, RDS db.r6g.large, 1 nodo Redis |
| 1 ano | 50.000 | 500 | Auto-scaling 2-8 tasks, read replica RDS, Redis 3 nodos |
| 3 anos | 200.000 | 2.000 | Evaluar microservicios (ver nota CM-001), RDS db.r6g.2xlarge, Redis 6 nodos |

Trigger de escalado: CPU > 65% o request count > 1.000/min por mas de 2 minutos.

## 3. Seguridad (STRIDE)

> Referencia: Microsoft STRIDE Threat Modeling

| Amenaza | Riesgo | Mitigacion |
|---|---|---|
| **Spoofing** (suplantacion de identidad) | Alto | JWT RS256 con expiracion 15min + refresh token rotativo 24h + MFA obligatorio en operaciones financieras (compra CDT, retiro). Tokens firmados con clave asimetrica rotada trimestralmente |
| **Tampering** (manipulacion de datos) | Critico | TLS 1.3 obligatorio en todas las comunicaciones. Audit trail inmutable con hash encadenado SHA-256. Checksums en mensajes SQS. Validacion de integridad en cada transaccion financiera |
| **Repudiation** (repudio de acciones) | Critico | Audit trail completo con timestamp UTC, userId, IP origen, user-agent, accion, y hash encadenado. Logs inmutables en CloudWatch Logs con retention policy de 7 anos (requisito SFC) |
| **Information Disclosure** (fuga de datos) | Alto | Cifrado en reposo AES-256 (RDS, S3, ElastiCache). Cifrado en transito TLS 1.3. RLS (Row-Level Security) en PostgreSQL: cada usuario solo accede a sus propios datos. PII enmascarado en logs |
| **Denial of Service** (denegacion de servicio) | Medio | AWS WAF con reglas de rate limiting (200 req/min por IP, 50 req/min por usuario autenticado). CloudFront como CDN y capa de absorcion. Auto-scaling reactivo en ECS |
| **Elevation of Privilege** (elevacion de privilegios) | Alto | MFA obligatorio para operaciones financieras. RBAC con principio de minimo privilegio. RLS en PostgreSQL. Segregacion de roles (admin/operador/usuario). JWT claims validados en cada request |

### Flujo de seguridad
```
Internet -> CloudFront (TLS 1.3) -> AWS WAF (rate limiting, geo-blocking)
    -> ALB (TLS termination) -> ECS Fargate (VPC privada)
                                  |
                    JWT validation + MFA verification
                                  |
                    RDS (RLS) / ElastiCache / SQS
                    (subredes privadas, sin acceso publico)
                    Cifrado en reposo AES-256
```

## 4. Dependencias Externas

| Sistema | SLA contratado | Timeout | Fallback |
|---|---|---|---|
| Deceval (custodia de titulos) | 99.5% | 15 seg | Enqueue reintento en T+5min con backoff exponencial. Usuario ve estado "En Proceso". Alerta P2 si 3 reintentos fallan |
| PSE (pagos electronicos) | 99.9% | 30 seg | Reintento automatico x2 con backoff. Si falla, notificar usuario via push + email. Alerta P1 si tasa de error > 10% |
| Core Bancario (saldos, cuentas) | 99.9% | 10 seg | Cache de saldos en Redis (TTL 30s). Si core no responde, bloquear nuevas inversiones y mostrar ultimo saldo conocido con disclaimer |
| SFC (reportes regulatorios) | Sin SLA real-time (batch) | 60 seg | Reportes batch diarios via SFTP. Cola de reportes pendientes si falla. No impacta operacion en tiempo real |

## 5. Observabilidad (SA define QUE medir y umbrales; SWA/Ops implementa)

**Plataforma elegida:** AWS CloudWatch + X-Ray (ver ADR-001)

### Metricas criticas que SA define

| Metrica | Umbral de alarma | Por que importa |
|---|---|---|
| Disponibilidad API | < 99.5% en ventana de 1 hora | SLO comprometido, activa consumo de error budget |
| Latencia P99 consulta portafolio | > 200ms por 5 min | NFR violado, degradacion directa de UX |
| Latencia P99 compra CDT | > 500ms por 5 min | NFR violado, riesgo de timeout en flujo financiero |
| Tasa de error API (5xx) | > 1% en ventana de 5 min | Posible incidente, impacto en confianza del usuario |
| SQS queue depth | > 300 mensajes por 3 min | Backlog excesivo, procesamiento E2E > 5 min |
| Deceval error rate | > 15% en 10 min | Proveedor posiblemente degradado, riesgo de no registrar custodia |
| Tasa de conversion KYC | < 65% en 1 dia | Posible problema en flujo de onboarding o proveedor KYC |
| Error budget consumido | > 80% del mensual | Congelar deployments de features, priorizar estabilidad |

### Lo que SWA/Ops implementa (fuera de scope SA)
- Formato de logs (JSON fields, structured logging)
- Instrumentacion de trazas distribuidas con X-Ray SDK
- Configuracion de dashboards y alertas en CloudWatch
- Retencion y rotacion de logs (minimo 7 anos para audit trail por SFC)

## 6. SPOFs (Single Points of Failure)

| SPOF | Impacto | Mitigacion actual | Pendiente |
|---|---|---|---|
| Deceval (custodia) | No se pueden registrar titulos valores. Inversiones quedan en estado "En Proceso" indefinidamente | Reintento automatico con backoff exponencial. Cola de pendientes. Alerta P2 | Negociar SLA formal >= 99.5%. Evaluar custodio alternativo para DR |
| PSE (pagos) | No se pueden debitar fondos para compra de CDTs ni acreditar rendimientos | Reintento auto x2. Notificacion proactiva al usuario. Alerta P1 si > 10% error rate | Evaluar pasarela de pagos alternativa (Wompi/Addi) como fallback |
| Core Bancario | No se pueden validar saldos ni ejecutar movimientos contables | Cache de saldos (TTL 30s). Bloqueo preventivo de nuevas inversiones | Implementar circuit breaker con Hystrix/Resilience4j. Modo offline con reconciliacion |
| RDS Primary (PostgreSQL) | Indisponibilidad total del sistema. Perdida de transacciones en curso | Multi-AZ con failover automatico (< 60s). Backups continuos c/hora | Testear failover trimestralmente. Documentar procedimiento de recovery |
| Redis (cache de sesiones/portafolio) | Degradacion de latencia. Consultas van directo a RDS, posible sobrecarga | Cluster mode Multi-AZ. Fallback a RDS con timeout ajustado | Implementar cache L1 local (Caffeine) para datos mas consultados |

## 7. Capacity Planning (SA define umbrales y estrategia; Ops ejecuta)

### SA define:

| Recurso | Actual (1K usuarios) | Proyeccion 6m (25K) | Proyeccion 1a (50K) | Trigger de escalado | Estrategia |
|---|---|---|---|---|---|
| ECS Tasks | 2 (0.5 vCPU, 1GB cada) | 4 | 8 | CPU > 65% por 2 min o request count > 1K/min | Auto-scaling horizontal |
| RDS instance | db.r6g.large (2 vCPU, 16GB) | db.r6g.large + read replica | db.r6g.xlarge (4 vCPU, 32GB) + read replica | CPU > 75% por 5 min | Vertical + read replicas |
| RDS storage | 50 GB (gp3) | 200 GB | 500 GB | > 80% usado | Storage auto-scaling ON (max 1TB) |
| Redis | 1 nodo cache.r6g.large | 3 nodos cluster | 6 nodos cluster | Memory > 75% o evictions > 0 | Horizontal (cluster scaling) |
| SQS | Ilimitado (managed) | Ilimitado | Ilimitado | N/A | Managed AWS |
| CloudFront | Pay-per-use | Pay-per-use | Pay-per-use | N/A | Managed AWS |

### Ops implementa:
- Configuracion de auto-scaling rules en ECS y ElastiCache
- Monitoreo de metricas de capacidad en CloudWatch
- Ejecucion de scaling manual cuando aplica (RDS vertical)
- Pruebas de carga trimestrales para validar proyecciones

## 8. Context Map (DDD)

> Ver detalle completo en CM-001.

Los 5 bounded contexts (Portafolio, KYC/Identidad, Custodia, Pagos, Compliance) se
integran segun patrones DDD. Las relaciones criticas:
- Portafolio -> Custodia: **Anti-Corruption Layer** (protege dominio core de API legacy Deceval)
- Pagos -> PSE: **Conformist** (nos adaptamos al protocolo nacional sin negociacion)
- Portafolio -> Compliance: **Published Language** (eventos de dominio estandarizados)
- KYC/Identidad -> Portafolio: **Customer-Supplier** (Identidad provee usuario verificado)

**Nota de implementacion:** Monolito modular para MVP. Evaluar microservicios si > 200K usuarios (ver CM-001).
