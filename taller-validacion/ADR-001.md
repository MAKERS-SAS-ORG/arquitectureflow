# ADR-001: Patron event-driven con SQS FIFO para integraciones externas

> **Origen:** CB-001 (restriccion: equipo .NET + AWS, integraciones Deceval/PSE),
> RFC-001 seccion 5.2 (riesgo de latencia de proveedores externos)

## Estado
Aceptado

## Contexto y Problema

La plataforma de micro-inversion en CDTs (RFC-001, opcion C aprobada) necesita orquestar
operaciones financieras que involucran cuatro sistemas externos identificados en CB-001:
Deceval (custodia de valores), PSE/ACH (pagos), core bancario (KYC) y SFC (reportes).

El problema central es que estas integraciones tienen latencias incompatibles con una
experiencia de usuario sincrona:
- **Deceval (custodia):** 1-30 segundos por operacion de registro/consulta
- **PSE (pagos):** 2-15 segundos para confirmar debito/abono
- **Core bancario (KYC):** 1-5 segundos para validacion de identidad
- **Flujo completo de compra de CDT:** potencialmente 4-50 segundos end-to-end

Una arquitectura sincrona request-response obligaria al usuario a esperar hasta 50 segundos
con una pantalla de carga, lo cual es inaceptable para una experiencia de micro-inversion
que busca ser simple y rapida. Adicionalmente, si cualquier sistema externo falla a mitad
del flujo, el manejo de compensaciones sincronas es fragil y propenso a inconsistencias.

## Decision Drivers
- **Latencia de proveedores externos:** Deceval 1-30s, PSE 2-15s — sincrono E2E es inaceptable
- **Confiabilidad financiera:** Las operaciones MUST completarse o compensarse, nunca quedar en limbo
- **Auditabilidad:** Ley 964/2005 exige trazabilidad completa de cada operacion financiera
- **Idempotencia:** Reintentos no deben duplicar transacciones financieras
- **Skills del equipo:** 5 ingenieros con experiencia en .NET y AWS (CB-001)
- **Volumen esperado:** 300 inversiones/dia, 500 operaciones/hora en pico (CB-001)

## Opciones Consideradas
- Opcion 1: API sincrona request-response end-to-end
- Opcion 2: Event-driven con Amazon SQS FIFO + Workers
- Opcion 3: Full event sourcing con CQRS y EventStore

## Decision

**Opcion elegida: "Event-driven con Amazon SQS FIFO + Workers"**

### Justificacion
Porque la naturaleza asincrona de las operaciones financieras (Deceval 1-30s, PSE 2-15s)
se alinea directamente con el patron event-driven: el API REST recibe la solicitud del
usuario, responde inmediatamente con 202 Accepted y un ID de operacion, y delega el
procesamiento a Workers que consumen mensajes de colas SQS FIFO.

Se elige SQS FIFO especificamente porque:
1. **Orden garantizado:** Las operaciones financieras de un mismo usuario deben procesarse
   en orden (no se puede redimir un CDT antes de confirmarse la compra)
2. **Exactly-once processing:** Elimina el riesgo de duplicar transacciones financieras
3. **Dead Letter Queue nativa:** Operaciones fallidas se envian a DLQ para reintento manual
   o automatizado, sin perder la transaccion
4. **Managed service en AWS:** El equipo ya tiene experiencia con AWS (CB-001), sin overhead
   operativo de mantener infraestructura de mensajeria

A diferencia de la Opcion 1 (sincrona) que bloquearia al usuario hasta 50 segundos y no
tiene mecanismo natural de reintento ante fallos de proveedores. Y de la Opcion 3 (full
event sourcing + CQRS) que introduce complejidad desproporcionada para 300 operaciones/dia:
EventStore, proyecciones, snapshots, y un modelo de consistencia que el equipo de 5
personas no tiene experiencia para operar.

## Pros y Contras de las Opciones

### Opcion 1: API sincrona request-response end-to-end
- (+) Simplicidad de implementacion y debugging
- (+) Consistencia inmediata: el usuario ve el resultado al instante
- (+) Sin infraestructura adicional de mensajeria
- (-) Usuario bloqueado 4-50 segundos esperando respuesta de proveedores
- (-) Timeout del gateway si Deceval tarda > 30s
- (-) Sin mecanismo natural de reintento: fallo = operacion perdida
- (-) Compensaciones manuales ante fallos parciales (debito exitoso pero custodia fallo)

### Opcion 2: Event-driven con Amazon SQS FIFO + Workers
- (+) Desacoplamiento temporal: usuario recibe 202 Accepted en < 500ms
- (+) Reintentos automaticos con backoff via SQS retry policy
- (+) DLQ para operaciones fallidas: ninguna transaccion se pierde
- (+) Orden garantizado por MessageGroupId (operaciones del mismo usuario en secuencia)
- (+) Exactly-once processing elimina riesgo de duplicacion financiera
- (+) Audit trail implicito: cada mensaje es un registro de la operacion
- (+) Servicio gestionado AWS: sin overhead operativo de mantener broker
- (-) Consistencia eventual: el usuario ve "En Proceso" antes de "Confirmada"
- (-) Requiere componente Worker adicional (proceso de background)
- (-) Debugging de flujos distribuidos requiere correlacion de IDs
- (-) Limite de throughput: 300 mensajes/segundo por cola FIFO (suficiente para 500 ops/hora)

### Opcion 3: Full event sourcing con CQRS y EventStore
- (+) Auditabilidad perfecta: cada cambio de estado es un evento inmutable
- (+) Capacidad de reconstruir estado desde el origen de eventos
- (+) Separacion de modelo de lectura y escritura optimiza cada caso
- (-) Complejidad operativa alta: EventStore, proyecciones, snapshots, rebuild
- (-) Equipo de 5 personas sin experiencia en event sourcing
- (-) Overhead desproporcionado para 300 operaciones/dia
- (-) Consistencia eventual mas compleja que con colas simples
- (-) Curva de aprendizaje de 2-3 meses antes de ser productivos

## Consecuencias

### Positivas
- El usuario recibe confirmacion inmediata (202 Accepted + ID de operacion) sin depender
  de la latencia de Deceval o PSE
- Operaciones fallidas se capturan en DLQ: el equipo de operaciones puede investigar y
  reintentar sin perder la transaccion del usuario
- Audit trail implicito en cada mensaje satisface requisitos de trazabilidad de Ley 964/2005
- El patron es extensible: agregar nuevas integraciones (ej: proveedor KYC alternativo)
  solo requiere un nuevo Worker consumidor

### Negativas / Trade-offs
- **Consistencia eventual:** El usuario vera "En Proceso" durante 4-50 segundos antes
  de ver "Confirmada". Requiere UX clara con indicadores de estado y notificaciones
- **Componente Worker:** Se necesita un proceso de background que consuma mensajes y
  orqueste las llamadas a Deceval, PSE y core bancario. Aumenta la superficie operativa
- **Idempotencia obligatoria:** Cada operacion MUST tener un idempotency key unico para
  garantizar que reintentos no dupliquen transacciones financieras (futuro ADR-002)
- **Monitoreo de colas:** Se necesitan alarmas sobre profundidad de cola, mensajes en DLQ,
  y latencia de procesamiento de Workers

### Neutrales
- El patron no define la tecnologia del Worker: puede ser AWS Lambda, ECS Task, o proceso
  .NET en EC2. Esa decision se tomara en el System Design
- Requiere estrategia de correlacion de IDs entre API, SQS y proveedores externos
- El limite de 300 msgs/s de SQS FIFO es 2.160x el volumen pico esperado (500 ops/hora)

## Validacion
- **Metrica:** Tiempo promedio de procesamiento de compra de CDT < 2 minutos E2E
- **Metrica:** 0 transacciones duplicadas en produccion (medido via idempotency checks)
- **Metrica:** < 0.1% de operaciones en DLQ sin resolucion despues de 24 horas
- **Senal de alerta:** Cola con profundidad > 200 mensajes sostenida por mas de 5 minutos
- **Senal de alerta:** Incremento de mensajes en DLQ > 5 en una hora
- **Revision programada:** 6 meses post-lanzamiento — evaluar si el volumen justifica
  migrar a event sourcing (Opcion 3) o si SQS FIFO sigue siendo adecuado
