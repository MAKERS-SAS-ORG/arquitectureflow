# Architecture Fitness Functions: Plataforma de Micro-Inversion en CDTs

> Referencia: Ford, N.; Parsons, R.; Kua, P. *Building Evolutionary Architectures.* 2nd Ed. O'Reilly, 2023.
>
> **Fitness Functions como contrato:** Las definiciones del SA (latencia < 500ms,
> audit trail completo, zero vulnerabilidades) se convierten en pruebas automatizadas
> que el equipo de ingenieria implementa en CI/CD y monitoring. Asi las decisiones
> del SA no se quedan en documentos — se verifican objetivamente en cada deploy
> y durante toda la vida del sistema.
>
> Cada FF traza a un NFR del PRD-001, una restriccion del CB-001, o una decision del CM-001.

## Clasificacion por Modo de Ejecucion

| Modo | Que valida | Cuando corre | Herramientas |
|---|---|---|---|
| **Estatica** | Reglas estructurales en el codigo (capas, dependencias, convenciones) | Build time / CI en cada PR | NetArchTest, ArchUnit, Snyk, SAST |
| **Dinamica** | Comportamiento real bajo carga o en produccion (latencia, throughput, disponibilidad) | CI con load test / Monitoring continuo | k6, JMeter, CloudWatch, Datadog |
| **Juicio** | Coherencia y vigencia de artefactos arquitectonicos (.md) | Sprint review / Bajo demanda | Agente IA (Claude, Copilot) |

> **Juicio con Agente IA:** El agente lee los artefactos .md del proyecto (CB, RFC, ADR, Tech Spec,
> System Design, FF) y evalua: consistencia entre artefactos, TODOs criticos sin resolver,
> decisiones que ya no aplican, NFRs sin fitness function, y desviaciones no documentadas.
> No reemplaza las FF estaticas/dinamicas — las complementa con analisis semantico.

## Catalogo de Fitness Functions

### FF-001: Latencia de API bajo carga
| Campo | Valor |
|---|---|
| Atributo de calidad | Performance |
| **Traza a** | **QA-001 (PRD-001): latencia < 500ms p99** |
| Tipo | Atomic |
| Modo | **Dinamica** |
| Trigger | CI Pipeline (en cada merge a main) |
| Metrica | P99 latencia de endpoints criticos (POST /inversiones, GET /portafolio, GET /rendimientos) |
| Umbral | < 500ms bajo 100 req/seg simulados |
| Herramienta | k6 load test en pipeline CI |
| Accion si falla | Bloquear deploy |

### FF-002: Zero vulnerabilidades criticas
| Campo | Valor |
|---|---|
| Atributo de calidad | Security |
| **Traza a** | **QA-003 (PRD-001): zero accesos no autorizados + SD-001 STRIDE** |
| Tipo | Atomic |
| Modo | **Estatica** |
| Trigger | CI Pipeline (en cada PR) |
| Metrica | Vulnerabilidades criticas y altas en dependencias y codigo fuente |
| Umbral | 0 criticas, 0 altas |
| Herramienta | Snyk (dependencias NuGet) + SAST (analisis de codigo) en pipeline CI |
| Accion si falla | Bloquear merge |

### FF-003: Integridad del audit trail
| Campo | Valor |
|---|---|
| Atributo de calidad | Compliance |
| **Traza a** | **RN-003 (PRD-001): audit trail inmutable + CB-001 compliance (retencion 10 anos)** |
| Tipo | Holistic (involucra API + Worker + DB) |
| Modo | **Dinamica** |
| Trigger | Deployment (test de integracion post-deploy a staging) |
| Metrica | Toda operacion financiera tiene entrada en audit trail con timestamp, actor, accion y hash de integridad valido |
| Umbral | 100% de operaciones auditadas — cero operaciones sin registro |
| Herramienta | Test de integracion automatizado que ejecuta flujo E2E (compra CDT completa) y verifica que cada paso genero entrada en audit trail |
| Accion si falla | Bloquear deploy a produccion, escalar a SA |

### FF-004: Aislamiento de datos por usuario
| Campo | Valor |
|---|---|
| Atributo de calidad | Security |
| **Traza a** | **RN-002 (PRD-001): usuario solo ve sus datos + STRIDE Information Disclosure (SD-001)** |
| Tipo | Atomic |
| Modo | **Dinamica** |
| Trigger | CI Pipeline (en cada merge a main) |
| Metrica | Ningun endpoint retorna datos de un usuario diferente al autenticado |
| Umbral | 0 violaciones de aislamiento |
| Herramienta | Test automatizado con multiples tokens JWT de usuario — cada uno ejecuta operaciones y verifica que solo accede a sus propios datos (portafolio, transacciones, rendimientos) |
| Accion si falla | Bloquear deploy, incidente P1 |

### FF-005: Disponibilidad del flujo de compra
| Campo | Valor |
|---|---|
| Atributo de calidad | Reliability |
| **Traza a** | **QA-002 (PRD-001): disponibilidad 99.5%, tiempo E2E < 5 min** |
| Tipo | Holistic (API + Worker + Deceval + PSE) |
| Modo | **Dinamica** |
| Trigger | Continuo (monitoring en produccion) |
| Metrica | Porcentaje de inversiones que completan E2E (solicitud -> confirmacion custodia) en < 5 min |
| Umbral | >= 99.5% disponibilidad en ventana de 30 dias |
| Herramienta | CloudWatch custom metric + alarma configurada con threshold 99.5% |
| Accion si falla | Alarma P2, investigar cuellos de botella en Worker, Deceval o PSE |

### FF-006: Desacoplamiento entre bounded contexts
| Campo | Valor |
|---|---|
| Atributo de calidad | Maintainability (Coupling) |
| **Traza a** | **CM-001: bounded contexts con patron ACL/Published Language** |
| Tipo | Atomic |
| Modo | **Estatica** |
| Trigger | CI Pipeline (en cada PR) |
| Metrica | Ningun bounded context importa implementaciones internas de otro (solo contratos publicos/interfaces) |
| Umbral | 0 violaciones de dependencia directa entre bounded contexts |
| Herramienta | NetArchTest en pipeline CI — reglas: `Portafolio` no referencia internals de `Custodia`, `Identidad` no referencia internals de `Inversiones`, etc. |
| Accion si falla | Bloquear merge |

### FF-007: Coherencia de artefactos arquitectonicos
| Campo | Valor |
|---|---|
| Atributo de calidad | Governance |
| **Traza a** | **TAA-001: consistencia entre artefactos y vigencia de decisiones** |
| Tipo | Holistic |
| Modo | **Juicio** |
| Trigger | Sprint Review / Bajo demanda del SA |
| Metrica | Agente IA analiza los artefactos .md del proyecto y evalua: (1) TODOs criticos sin resolver, (2) ADRs que contradicen TS-001 o entre si, (3) NFRs del PRD sin FF asociada, (4) Desviaciones en TAA sin justificacion, (5) Artefactos en Draft que deberian estar en Review |
| Umbral | 0 inconsistencias criticas entre artefactos, 0 TODOs bloqueantes |
| Herramienta | Agente IA (Claude Code / Copilot) con acceso a la carpeta de artefactos |
| Accion si falla | SA revisa inconsistencias, actualiza artefactos afectados, registra en TAA seccion 4 si hay desviacion |

**Prompt de ejemplo para FF-007 (Juicio):**
```
Lee todos los artefactos .md en esta carpeta (CB-001, RFC-001, ADR-001, ADR-002,
PRD-001, TS-001, SD-001, RO-001, CM-001, FF-001, TAA-001).

Evalua:
1. Hay TODOs criticos sin resolver en artefactos Approved?
2. Alguna decision en los ADRs contradice lo definido en TS-001?
3. Hay NFRs en PRD-001 que no tienen Fitness Function asociada?
4. Hay desviaciones en TAA-001 seccion 4 sin justificacion?
5. Hay artefactos en Draft que ya deberian estar en Review dado el estado del proyecto?

Responde: el sistema sigue siendo arquitectonicamente apto (fit)?
Clasifica cada hallazgo como: CRITICO / IMPORTANTE / SUGERENCIA.
```

### Prioridad de implementacion recomendada

1. **FF-002 + FF-006** (Estatica) — se implementan primero porque son las mas faciles de configurar: solo necesitan Snyk y NetArchTest en el pipeline CI desde Sprint 0
2. **FF-001 + FF-004** (Dinamica CI) — segundo, agregar k6 load test y test de aislamiento multi-token al pipeline en Sprint 1
3. **FF-003** (Dinamica deploy) — tercero, test de integracion post-deploy que valida audit trail E2E en Sprint 1-2
4. **FF-005** (Dinamica monitoring) — cuarto, configurar metrica custom de CloudWatch cuando se despliegue a produccion
5. **FF-007** (Juicio) — quinto, ejecutar manualmente con agente IA en el primer sprint review
