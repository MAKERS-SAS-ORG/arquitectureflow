## 2. Requisitos de Deployment

| Requisito | Valor |
|---|---|
| Estrategia | Blue-green deployment via ECS con ALB target group switching |
| Tiempo maximo de deployment | 15 minutos (incluyendo bake time de 5 min) |
| Health check endpoint | GET /health |
| Health check criterio | HTTP 200 con body `{"status":"healthy","db":"connected","cache":"connected","queue":"connected"}` en < 300ms |
| Bake time post-switch | 5 minutos observando metricas antes de terminar grupo blue |
| Imagen de rollback | Imagen anterior retenida en ECR por 72 horas minimo |

### Checklist de verificacion post-despliegue (SA define criterios)

- [ ] GET /health retorna 200 en los 3 target groups (Multi-AZ)
- [ ] Tasa de error 5xx < 0.5% en los primeros 5 minutos post-deploy
- [ ] Flujo principal funciona: crear inversion de prueba en staging, verificar confirmacion E2E
- [ ] Audit trail registra la operacion de prueba con hash encadenado valido
- [ ] SQS queue depth estable (no creciendo, delta < 50 mensajes en 5 min)
- [ ] Latencia P99 del portafolio < 200ms (dentro del SLO definido en SD-001)

## 3. Criterios de Rollback

| Condicion | Accion |
|---|---|
| Tasa de error 5xx > 5% por mas de 5 minutos | Rollback automatico: switch ALB al target group anterior (blue) |
| Health check falla en cualquier AZ por 60 segundos post-deploy | Rollback automatico: revertir target group en ALB |
| Latencia P99 > 1s por mas de 5 minutos | Evaluacion manual por on-call, rollback probable |
| SQS DLQ recibe > 20 mensajes en 10 minutos | Investigar causa raiz, rollback si correlaciona con deploy |
| Audit trail deja de registrar eventos por > 2 minutos | Rollback automatico: riesgo de compliance SFC |

**Ventana de rollback segura:** 30 minutos post-deploy (ECS mantiene imagen anterior activa en target group blue).
Despues de 30 minutos, rollback requiere redeploy con imagen anterior desde ECR.

## 4. Metricas Criticas y Umbrales

| Metrica | Umbral de alarma | Severidad | Escalar a |
|---|---|---|---|
| Disponibilidad API | < 99.5% en ventana de 1 hora | P1 | Tech Lead + On-call + Arquitecto SA |
| Tasa de error API (5xx) | > 1% en ventana de 5 min | P1 | Tech Lead + On-call |
| Latencia P99 consulta portafolio | > 200ms por 5 min | P2 | On-call |
| Latencia P99 compra CDT | > 500ms por 5 min | P2 | On-call |
| SQS queue depth | > 300 mensajes por 3 min | P2 | On-call |
| Deceval error rate | > 15% por 10 min | P2 | On-call + Tech Lead |
| RDS CPU | > 75% por 5 min | P2 | On-call |
| Redis memory utilization | > 75% o evictions > 0 | P2 | On-call |
| DLQ message count | > 0 por cualquier duracion | P3 | On-call (proximo dia habil si < 5 mensajes) |
| Error budget consumido | > 80% del mensual (> 2.88h de downtime) | P1 | Tech Lead + Arquitecto SA + CTO |

### Niveles de severidad

| Nivel | Definicion | Tiempo de respuesta | Ejemplo |
|---|---|---|---|
| P1 | Sistema caido, datos financieros en riesgo, SLO roto, o compliance comprometido | < 15 minutos | API no responde, audit trail detenido, error budget agotado |
| P2 | Funcionalidad degradada, operaciones lentas, proveedor externo con problemas | < 1 hora | Latencia elevada, Deceval con errores intermitentes, CPU alta |
| P3 | Issue menor, mensajes en DLQ, metricas anomalas sin impacto directo | Proximo dia habil | DLQ con pocos mensajes, metrica cerca del umbral |

## 5. Dependencias Externas y SLAs

| Sistema externo | SLA contratado | Timeout esperado | Comportamiento si falla |
|---|---|---|---|
| Deceval (custodia) | 99.5% | 15 seg | Reintento auto en T+5min con backoff exponencial. Usuario ve "Inversion en Proceso". Alerta P2 si 3 reintentos fallan consecutivamente |
| PSE (pagos electronicos) | 99.9% | 30 seg | Reintento automatico x2 con backoff. Si falla, notificar usuario via push notification y email. Alerta P1 si tasa de error > 10% en 5 min |
| Core Bancario | 99.9% | 10 seg | Cache de saldos en Redis (TTL 30s). Bloquear nuevas inversiones si core no responde. Mostrar ultimo saldo con disclaimer. Alerta P1 si caida > 5 min |
| SFC (reportes regulatorios) | Sin SLA real-time (batch diario) | 60 seg | Cola de reportes pendientes. Reintento en siguiente ventana batch. No impacta operacion en tiempo real. Alerta P3 si falla 2 dias consecutivos |
| Proveedor KYC | 99% | 10 seg | Cola de verificacion manual. Usuario notificado "Verificacion en proceso, le notificaremos en 24h". Alerta P2 si > 20% de verificaciones fallan |

## 6. Politica de Gestion de Cambios

Todo cambio en produccion MUST:
- Tener ticket en Jira aprobado con descripcion del cambio y justificacion
- Tener plan de rollback documentado y probado en staging
- Ejecutarse en ventana de bajo trafico: martes a jueves, 6:00-8:00 AM COT
- No coincidir con ventanas de reporte SFC (ultimo dia habil del mes)
- Pasar todos los tests automatizados y verificacion en staging

Cambios que requieren aprobacion adicional del SA:
- Cambios en schema de base de datos (migraciones)
- Cambios en contratos de API (endpoints, payloads, versiones)
- Cambios en integraciones con Deceval, PSE, Core Bancario o proveedor KYC
- Actualizaciones de dependencias mayores (runtime, framework, base de datos)
- Cualquier cambio que afecte el audit trail o mecanismo de hash encadenado
- Cambios en reglas de WAF, rate limiting o politicas de seguridad

Cambios que requieren aprobacion de Compliance:
- Modificacion de flujo de KYC
- Cambios en estructura de reportes SFC
- Alteracion de politicas de retencion de datos
