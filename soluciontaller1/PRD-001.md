# PRD: Plataforma de Micro-Inversion en Renta Fija

## 1. Objetivo del Producto

Permitir que personas naturales colombianas inviertan en CDTs de renta fija desde
$100.000 COP a traves de un proceso 100% digital, capturando un mercado de 8 millones
de personas desatendidas.

## 2. Usuarios Objetivo

| Persona | Descripcion | Necesidad Principal |
|---|---|---|
| Inversionista Nuevo | Persona natural colombiana, 25-45 anos, ingreso < $5M COP/mes, sin experiencia en inversion | Invertir de forma simple y segura con montos bajos |
| Inversionista Recurrente | Usuario existente que ya hizo al menos una inversion | Consultar rendimientos y reinvertir al vencimiento |

## 3. Funcionalidades (Casos de Uso)

### CU-001: Onboarding con KYC

- **Actores:** Inversionista Nuevo, Sistema KYC (Core Bancario)
- **Precondicion:** Usuario no registrado en la plataforma
- **Flujo principal:**
  1. El usuario ingresa datos personales (nombre, cedula, email, telefono)
  2. El sistema valida identidad contra Core Bancario (KYC)
  3. El sistema verifica que no existe en listas restrictivas (SARLAFT)
  4. El sistema crea cuenta y notifica al usuario
- **Flujos alternativos:**
  - Alt-A: KYC rechazado → Notificar usuario, no crear cuenta, registrar en audit trail
  - Alt-B: Core bancario no responde en 30s → Mostrar error, reintentar
- **Postcondicion:** Usuario registrado con KYC aprobado, listo para invertir
- **Prioridad:** Must Have

### CU-002: Compra de CDT

- **Actores:** Inversionista (registrado), Deceval, PSE
- **Precondicion:** Usuario con KYC aprobado y fondos disponibles
- **Flujo principal:**
  1. El usuario selecciona tipo de CDT (emisor, plazo, tasa)
  2. El usuario ingresa monto (minimo $100.000 COP)
  3. El sistema valida reglas de negocio (monto, limites)
  4. El sistema registra la intencion de compra en audit trail
  5. El sistema inicia transferencia via PSE/ACH
  6. El sistema registra custodia en Deceval
  7. El sistema confirma al usuario y actualiza portafolio
- **Flujos alternativos:**
  - Alt-A: PSE rechaza pago → Cancelar operacion, notificar usuario
  - Alt-B: Deceval no responde → Encolar en SQS (ADR-001), reintentar con backoff
  - Alt-C: Monto menor a $100.000 → Rechazar con mensaje "monto minimo $100.000 COP"
- **Postcondicion:** CDT registrado en Deceval, fondos transferidos, portafolio actualizado
- **Prioridad:** Must Have

### CU-003: Consulta de Portafolio

- **Actores:** Inversionista
- **Precondicion:** Usuario autenticado con al menos una inversion
- **Flujo principal:**
  1. El usuario accede a la seccion "Mi Portafolio"
  2. El sistema muestra lista de CDTs activos (emisor, monto, tasa, vencimiento)
  3. El sistema muestra rendimiento acumulado por CDT y total
- **Flujos alternativos:**
  - Alt-A: Sin inversiones → Mostrar mensaje "Aun no tienes inversiones" + CTA
- **Postcondicion:** Usuario ve estado actual de sus inversiones
- **Prioridad:** Must Have

### CU-004: Redencion Automatica al Vencimiento

- **Actores:** Sistema (automatico), Deceval, PSE
- **Precondicion:** CDT llega a fecha de vencimiento
- **Flujo principal:**
  1. Proceso batch detecta CDTs vencidos
  2. El sistema solicita redencion a Deceval
  3. Deceval confirma liquidacion (capital + rendimiento)
  4. El sistema transfiere fondos al usuario via PSE/ACH
  5. El sistema actualiza portafolio y notifica al usuario
- **Flujos alternativos:**
  - Alt-A: Deceval no confirma en 24h → Escalar a Operaciones, P2
- **Postcondicion:** Fondos transferidos al usuario, CDT cerrado en portafolio
- **Prioridad:** Must Have

## 4. User Stories con Criterios de Aceptacion

### US-001: Registro con KYC

**Como** inversionista nuevo, **quiero** registrarme verificando mi identidad,
**para** poder invertir de forma segura.

**Criterios de Aceptacion:**
```gherkin
Scenario: Registro exitoso con KYC aprobado
  Given un usuario no registrado con cedula valida
  When  ingresa sus datos y el sistema consulta KYC al Core Bancario
  Then  la cuenta se crea con estado "activo"
  And   el usuario recibe confirmacion por email
  And   el evento queda en audit trail

Scenario: KYC rechazado
  Given un usuario con cedula en lista restrictiva SARLAFT
  When  el sistema consulta KYC al Core Bancario
  Then  la cuenta NO se crea
  And   el usuario ve mensaje "No pudimos verificar tu identidad"
  And   el rechazo queda en audit trail con motivo
```

### US-002: Compra de CDT desde $100K

**Como** inversionista registrado, **quiero** comprar un CDT desde $100.000 COP,
**para** hacer crecer mis ahorros con renta fija.

**Criterios de Aceptacion:**
```gherkin
Scenario: Compra exitosa de CDT
  Given un usuario autenticado con KYC aprobado
  When  selecciona un CDT de $300.000 COP a 90 dias
  And   confirma la compra
  Then  ve mensaje "Tu inversion esta en proceso"
  And   recibe confirmacion cuando Deceval registra la custodia
  And   el CDT aparece en su portafolio

Scenario: Monto menor al minimo
  Given un usuario autenticado
  When  intenta comprar un CDT por $50.000 COP
  Then  ve mensaje "El monto minimo de inversion es $100.000 COP"
  And   la operacion no se procesa
```

## 5. Quality Attribute Scenarios (NFRs)

> Referencia: ISO/IEC 25010:2023, Bass et al. Quality Attribute Scenarios

### QA-001: Performance — Latencia de API

| Campo | Valor |
|---|---|
| Fuente | Usuario autenticado |
| Estimulo | Solicita compra de CDT o consulta de portafolio |
| Artefacto | API de Inversiones |
| Entorno | Carga normal (50 usuarios concurrentes, 100 req/seg) |
| Respuesta | Operacion registrada y confirmacion enviada |
| Medida | **Latencia < 500ms p99** para endpoints de la API |

### QA-002: Reliability — Disponibilidad del Flujo de Compra

| Campo | Valor |
|---|---|
| Fuente | Sistema (monitoring) |
| Estimulo | Verificacion de disponibilidad del flujo de compra E2E |
| Artefacto | Plataforma completa (API + Worker + Integraciones) |
| Entorno | Operacion normal |
| Respuesta | Inversiones completan flujo E2E sin intervencion manual |
| Medida | **Disponibilidad 99.5% mensual** (maximo 3.6h downtime/mes). **95% de inversiones completadas E2E en < 5 min** |

### QA-003: Security — Zero Accesos No Autorizados

| Campo | Valor |
|---|---|
| Fuente | Atacante externo o usuario malicioso |
| Estimulo | Intento de acceder a datos de otro usuario |
| Artefacto | API completa |
| Entorno | Operacion normal |
| Respuesta | Solicitud rechazada, intento registrado en audit trail |
| Medida | **Zero accesos no autorizados a datos de otros usuarios** |

## 6. Reglas de Negocio

- RN-001: Monto minimo de inversion: $100.000 COP por operacion
- RN-002: Un usuario solo puede ver y operar con sus propios datos (aislamiento por tenant)
- RN-003: Toda operacion financiera debe registrarse en audit trail ANTES de ejecutarse (fail-closed)
- RN-004: KYC obligatorio y aprobado antes de cualquier operacion de inversion
- RN-005: Redencion solo al vencimiento del CDT (no anticipada en MVP)
- RN-006: Solo inversiones en COP (sin multi-moneda)
- RN-007: Limite maximo de inversion por usuario en fase sandbox: $10.000.000 COP acumulado (validado con Compliance 2026-04-29)

## 7. Fuera de Scope (explicito)

- Renta variable (acciones) — regulacion diferente
- Mercado secundario de CDTs — fase futura
- App mobile nativa — MVP es web responsive
- Robo-advisor — requiere IA, fase futura
- Inversiones recurrentes automaticas — fase futura
- Redencion anticipada — simplifica MVP y regulacion

## 8. Dependencias y Restricciones

- Depende de: API KYC del Core Bancario, API Deceval v3.2, PSE/ACH
- Restricciones: Ley 964/2005, Ley 1581/2012, Sandbox SFC, equipo de 5, presupuesto $1.200M COP
