# Architecture Fitness Functions: Plataforma de Inversion en Renta Fija

> Referencia: Ford, N.; Parsons, R.; Kua, P. *Building Evolutionary Architectures.* 2nd Ed. O'Reilly, 2023.
>
> **Fitness Functions como contrato:** Las definiciones del SA (latencia < 500ms,
> audit trail completo, zero vulnerabilidades) se convierten en pruebas automatizadas
> que el equipo de ingenieria implementa en CI/CD y monitoring. Asi las decisiones
> del SA no se quedan en documentos — se verifican objetivamente en cada deploy
> y durante toda la vida del sistema.
>
> Cada FF traza a un NFR del PRD-001, una restriccion del CB-001, o una decision del CM-001.

## Clasificacion por Modo de Ejecucion

| Modo | Que valida | Cuando corre | Herramientas |
|---|---|---|---|
| **Estatica** | Reglas estructurales en el codigo | Build time / CI en cada PR | NetArchTest, ArchUnit, Snyk |
| **Dinamica** | Comportamiento real bajo carga o en produccion | CI con load test / Monitoring continuo | k6, JMeter, CloudWatch |
| **Juicio** | Coherencia y vigencia de artefactos arquitectonicos (.md) | Sprint review / Bajo demanda | Agente IA (Claude, Copilot) |

## Catalogo de Fitness Functions

### FF-001: Latencia de API bajo carga
| Campo | Valor |
|---|---|
| Atributo de calidad | Performance |
| **Traza a** | **QA-001 (PRD-001): latencia < 500ms p99** |
| Tipo | Atomic |
| Modo | **Dinamica** |
| Trigger | CI Pipeline (en cada merge a main) |
| Metrica | P99 latencia de endpoints criticos (POST /inversiones, GET /portafolio) |
| Umbral | < 500ms bajo 100 req/seg simulados |
| Herramienta | k6 load test en pipeline CI |
| Accion si falla | Bloquear deploy |

### FF-002: Zero vulnerabilidades criticas
| Campo | Valor |
|---|---|
| Atributo de calidad | Security |
| **Traza a** | **QA-003 (PRD-001): zero accesos no autorizados** |
| Tipo | Atomic |
| Modo | **Estatica** |
| Trigger | CI Pipeline (en cada PR) |
| Metrica | Vulnerabilidades criticas y altas en dependencias y codigo |
| Umbral | 0 criticas, 0 altas |
| Herramienta | Snyk (dependencias) + SAST (codigo) en pipeline |
| Accion si falla | Bloquear merge |

### FF-003: Integridad del audit trail
| Campo | Valor |
|---|---|
| Atributo de calidad | Compliance |
| **Traza a** | **RN-003 (PRD-001): audit trail inmutable + CB-001 compliance (retencion 10 anos)** |
| Tipo | Holistic (involucra API + Worker + DB) |
| Modo | **Dinamica** |
| Trigger | Deployment (test de integracion post-deploy) |
| Metrica | Toda operacion financiera tiene entrada en audit trail con timestamp y hash valido |
| Umbral | 100% de operaciones auditadas |
| Herramienta | Test de integracion automatizado que ejecuta flujo E2E y verifica audit trail |
| Accion si falla | Bloquear deploy, escalar a SA |

### FF-004: Aislamiento de datos por usuario
| Campo | Valor |
|---|---|
| Atributo de calidad | Security |
| **Traza a** | **RN-002 (PRD-001): usuario solo ve sus datos + STRIDE Information Disclosure (SD-001)** |
| Tipo | Atomic |
| Modo | **Dinamica** |
| Trigger | CI Pipeline |
| Metrica | Ningun endpoint retorna datos de un usuario diferente al autenticado |
| Umbral | 0 violaciones |
| Herramienta | Test automatizado con multiples tokens de usuario — cada uno solo debe ver sus datos |
| Accion si falla | Bloquear deploy, incidente P1 |

### FF-005: Disponibilidad del flujo de compra
| Campo | Valor |
|---|---|
| Atributo de calidad | Reliability |
| **Traza a** | **QA-002 (PRD-001): disponibilidad 99.5%, tiempo E2E < 5 min** |
| Tipo | Holistic |
| Modo | **Dinamica** |
| Trigger | Continuo (monitoring en produccion) |
| Metrica | Porcentaje de inversiones que completan E2E (solicitud → confirmacion Deceval) en < 5 min |
| Umbral | > 95% de inversiones completadas en tiempo |
| Herramienta | CloudWatch custom metric con alarma |
| Accion si falla | Alarma P2, investigar cuellos de botella en Worker o Deceval |

### FF-006: Desacoplamiento entre bounded contexts
| Campo | Valor |
|---|---|
| Atributo de calidad | Maintainability (Coupling) |
| **Traza a** | **CM-001: bounded contexts con patron ACL/Published Language** |
| Tipo | Atomic |
| Modo | **Estatica** |
| Trigger | CI Pipeline (en cada PR) |
| Metrica | Ningun bounded context importa implementaciones internas de otro (solo contratos publicos) |
| Umbral | 0 violaciones de dependencia directa |
| Herramienta | NetArchTest en pipeline — reglas: `Portafolio` no referencia internals de `Custodia`, etc. |
| Accion si falla | Bloquear merge |

### FF-007: Coherencia de artefactos arquitectonicos
| Campo | Valor |
|---|---|
| Atributo de calidad | Governance |
| **Traza a** | **TAA-001: consistencia entre artefactos y vigencia de decisiones** |
| Tipo | Holistic |
| Modo | **Juicio** |
| Trigger | Sprint Review / Bajo demanda del SA |
| Metrica | Agente IA analiza los artefactos .md del proyecto y evalua: (1) TODOs criticos sin resolver, (2) ADRs que contradicen TS-001, (3) NFRs del PRD sin FF asociada, (4) Desviaciones en TAA sin justificacion, (5) Artefactos en Draft que deberian estar en Review |
| Umbral | 0 inconsistencias criticas entre artefactos, 0 TODOs bloqueantes |
| Herramienta | Agente IA (Claude Code / Copilot) con acceso a la carpeta de artefactos |
| Accion si falla | SA revisa inconsistencias, actualiza artefactos afectados, registra en TAA seccion 4 si hay desviacion |

**Prompt de ejemplo para FF-007 (Juicio):**
```
Lee todos los artefactos .md en esta carpeta (CB-001, RFC-001, ADR-001,
PRD-001, TS-001, SD-001, RO-001, CM-001, FF-001, TAA-001).

Evalua:
1. Hay TODOs criticos sin resolver en artefactos Approved?
2. Alguna decision en ADR-001 contradice lo definido en TS-001?
3. Hay NFRs en PRD-001 que no tienen Fitness Function asociada?
4. Hay desviaciones en TAA-001 seccion 4 sin justificacion?
5. Hay artefactos en Draft que ya deberian estar en Review?

Responde: el sistema sigue siendo arquitectonicamente apto (fit)?
Clasifica cada hallazgo como: CRITICO / IMPORTANTE / SUGERENCIA.
```
