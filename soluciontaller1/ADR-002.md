# ADR-002: PostgreSQL como Base de Datos Principal

## Estado

Aceptado

## Contexto y Problema

La plataforma necesita una base de datos relacional para: usuarios, portafolios,
transacciones financieras y audit trail inmutable. Las opciones reales del equipo
son PostgreSQL y SQL Server, dado que trabajan en .NET 8 (CB-001).

La base de datos debe soportar:
- Row-Level Security (aislamiento de datos por usuario — RN-002 del PRD-001)
- Tabla append-only para audit trail (RN-003 del PRD-001)
- JSON para esquemas semi-estructurados (eventos de dominio del CM-001)
- Retencion de 10 anos (regulacion financiera colombiana — CB-001)

## Decision Drivers

- **Costo:** Presupuesto limitado a $1.200M COP (CB-001) — licencias impactan
- **Row-Level Security nativo:** RN-002 exige aislamiento por usuario a nivel de DB
- **Experiencia del equipo:** 5 ingenieros .NET, experiencia mixta en ambos motores
- **Compatibilidad AWS RDS:** Ambos disponibles en RDS, pero costos diferentes

## Opciones Consideradas

- Opcion 1: SQL Server en RDS
- Opcion 2: PostgreSQL en RDS

## Decision

**Opcion elegida: "PostgreSQL en RDS"**

### Justificacion

PostgreSQL ofrece Row-Level Security nativo (critico para FF-004), JSONB para
eventos de dominio, y costo 60% menor que SQL Server en RDS (license-included).
Entity Framework Core 8 soporta PostgreSQL via Npgsql con paridad de features.

## Pros y Contras de las Opciones

### Opcion 1: SQL Server en RDS

- (+) El equipo lo conoce mejor historicamente
- (+) Integracion nativa con .NET (sin paquete adicional)
- (-) Costo 2.5x mayor en RDS (license-included: ~$450 USD/mes vs ~$180 USD/mes para db.r6g.large)
- (-) Row-Level Security disponible pero mas complejo de configurar
- (-) Licenciamiento adicional si se escala a replicas de lectura

### Opcion 2: PostgreSQL en RDS — ELEGIDA

- (+) Row-Level Security nativo y bien documentado
- (+) JSONB para almacenar eventos de dominio sin schema rigido
- (+) 60% mas barato en RDS ($180 vs $450 USD/mes en instancia equivalente)
- (+) Open source — sin riesgo de licenciamiento al escalar
- (+) Entity Framework Core 8 + Npgsql es production-ready
- (-) Equipo necesita 1-2 semanas de familiarizacion con particularidades PostgreSQL
- (-) Tooling de debug ligeramente diferente (pgAdmin vs SSMS)

## Consecuencias

### Positivas

- Ahorro de ~$3.200 USD/ano en infra (reinvertible en el proyecto)
- RLS nativo simplifica implementacion de FF-004 (aislamiento de datos)
- JSONB permite almacenar domain events sin migraciones de schema

### Negativas / Trade-offs

- Sprint 0 incluira 2-3 dias de setup del equipo con PostgreSQL + Npgsql
- Ops debe aprender pgAdmin y particularidades de pg_dump para backups

### Neutrales

- Multi-AZ y backups automaticos funcionan igual en RDS para ambos motores

## Validacion

- **Metrica:** Tiempo de queries criticas (portafolio, transacciones) < 50ms p99
- **Senal de alerta:** Queries que requieran workarounds por diferencias PostgreSQL vs SQL Server
- **Revision programada:** Sprint 2 — evaluar si la curva de aprendizaje impacto la velocidad
