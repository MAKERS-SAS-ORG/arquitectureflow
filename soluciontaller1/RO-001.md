# Requisitos Operacionales: Plataforma de Inversion en Renta Fija

> Este documento define los REQUISITOS que Operations/SRE necesita para
> crear el runbook detallado. El SA define QUE operar; Operations define COMO.

## 1. Informacion del Sistema

| Item | Valor |
|---|---|
| URL produccion | `https://api.plataforma.com` |
| URL staging | `https://staging-api.plataforma.com` |
| Region/Zona | AWS us-east-1 (Multi-AZ) |
| Dashboard de monitoreo | CloudWatch Dashboard "inversiones-prod" |
| Canal de alertas | SNS → Slack #inversiones-ops + PagerDuty (P1) |
| Contacto de escalada | Tech Lead (L1), Arquitecto SA (L2), Director de Producto (L3) |

## 2. Requisitos de Deployment

| Requisito | Valor |
|---|---|
| Estrategia | Blue-green via ECS deployments |
| Tiempo maximo de deployment | 15 minutos (incluyendo health checks) |
| Health check endpoint | `GET /health` |
| Health check criterio | HTTP 200, JSON `{"status":"healthy","db":"ok","cache":"ok"}` |
| Ventana de deployment | Lunes a jueves, 10:00-16:00 COT (fuera de horario pico) |
| Ventana de cambios de infra | Sabados 02:00-06:00 COT |

### Checklist de verificacion post-despliegue (SA define criterios)

- [ ] Health check retorna HTTP 200 con db y cache "ok"
- [ ] Tasa de error 5xx < 1% en los primeros 10 minutos
- [ ] Flujo de compra de CDT funciona E2E en staging antes de produccion
- [ ] Latencia P99 < 500ms en los primeros 10 minutos
- [ ] Queue depth SQS estable (no creciendo)
- [ ] Audit trail registrando operaciones correctamente

## 3. Criterios de Rollback

| Condicion | Accion |
|---|---|
| Tasa de error 5xx > 5% por mas de 5 minutos | Rollback automatico (ECS deployment rollback) |
| Health check falla en 60 segundos post-deploy | Rollback automatico |
| Latencia P99 > 2x baseline por mas de 10 minutos | Evaluacion manual, rollback si no mejora |
| DLQ recibe mensajes que antes no fallaban | Investigar, posible rollback manual |
| Audit trail no registra operaciones | Rollback inmediato (P1) — compliance en riesgo |

**Ventana de rollback segura:** 1 hora post-deploy. Despues de 1 hora, rollback
requiere evaluacion manual por riesgo de datos inconsistentes.

## 4. Metricas Criticas y Umbrales

| Metrica | Umbral de alarma | Severidad | Escalar a |
|---|---|---|---|
| Disponibilidad API | < 99.5% en 30 dias | P1 | Tech Lead + SA |
| Latencia P99 | > 500ms por 5 min | P2 | Tech Lead |
| Tasa error 5xx | > 5% en 5 min | P1 | Tech Lead + SA + PagerDuty |
| Queue depth SQS | > 100 msgs por 10 min | P2 | Tech Lead |
| DLQ depth | > 0 msgs por 30 min | P2 | Tech Lead + Operaciones financieras |
| DLQ depth | > 10 msgs por 1 hora | P1 | SA + Operaciones financieras |
| CPU ECS > 80% | Por 5 min | P3 | Auto-scaling deberia actuar |
| RDS connections > 80% | Por 5 min | P2 | Tech Lead |
| Intentos acceso no autorizado | > 10 en 1 hora | P1 | SA + Security |
| Reporte SFC fallido | 2 dias consecutivos | P1 | SA + Compliance + Operaciones financieras |

### Niveles de severidad

| Nivel | Definicion | Tiempo de respuesta |
|---|---|---|
| P1 | Sistema caido, datos en riesgo, o compliance comprometida | < 15 minutos |
| P2 | Funcionalidad degradada, operaciones financieras retrasadas | < 1 hora |
| P3 | Issue menor, sin impacto critico | Proximo dia habil |

## 5. Dependencias Externas y SLAs

| Sistema externo | SLA contratado | Timeout esperado | Comportamiento si falla |
|---|---|---|---|
| Deceval | 99.5%, 1-30s | 30s | Mensaje queda en SQS, retry con backoff, DLQ si 3 fallos. Operacion pendiente |
| PSE/ACH | 99.9%, 2-15s | 20s | Cancelar operacion, notificar usuario, reembolso si fondos ya transferidos |
| Core Bancario (KYC) | 99.9%, 500ms-3s | 5s | Error al usuario: "Servicio temporalmente no disponible", no permite registro |
| SFC (batch) | N/A | N/A | Reintentar envio proximo ciclo (02:00 AM). Si falla 2 dias: P1 |

## 6. Politica de Gestion de Cambios

Todo cambio en produccion MUST:
- Tener ticket aprobado con descripcion del cambio y plan de rollback
- Ejecutarse en ventana de deployment (Lun-Jue 10:00-16:00 COT)
- Pasar por staging con prueba E2E del flujo de compra

Cambios que requieren aprobacion adicional del SA:
- Cambios en schema de base de datos (migraciones)
- Cambios en contratos de API (nuevos endpoints, cambios de formato)
- Cambios en integraciones con Deceval, PSE o Core Bancario
- Actualizaciones de dependencias mayores (.NET, PostgreSQL)
- Cambios en politicas de seguridad (JWT, WAF, rate limiting)
