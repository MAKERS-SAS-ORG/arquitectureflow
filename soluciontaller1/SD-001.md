## 2. Drivers de Arquitectura

### 2.1 Requisitos de Rendimiento

| Escenario | Latencia objetivo | Throughput | Medicion |
|---|---|---|---|
| Consulta de portafolio | < 200ms P99 | 100 req/seg | CloudWatch API Gateway latency |
| Compra de CDT (registro) | < 500ms P99 | 50 req/seg | CloudWatch custom metric |
| Compra de CDT (E2E con Deceval) | < 5 min P95 | 300/dia | Metrica custom: tiempo entre solicitud y confirmacion |
| KYC (consulta Core Bancario) | < 3s P99 | 20 req/seg | X-Ray trace duration |

### 2.2 Disponibilidad y Tolerancia a Fallos

| Componente | SLO | Estrategia |
|---|---|---|
| API de Inversiones | 99.5% mensual | ECS Fargate multi-AZ, min 2 tasks |
| Worker de Eventos | 99.5% mensual | ECS Fargate, auto-restart, DLQ como safety net |
| Base de datos | 99.95% | RDS Multi-AZ con failover automatico |
| Cache Redis | 99.9% | ElastiCache cluster multi-AZ |

**Error Budget:** 99.5% = maximo 3.6 horas de downtime por mes.
Si se consume el 80% del error budget (2.88h), congelar deploys no criticos.

**RPO:** 1 hora (backups RDS cada hora + logs de transacciones)
**RTO:** 30 minutos (failover automatico RDS + re-deploy ECS)

### 2.3 Escalabilidad

| Horizonte | Usuarios | Inversiones/dia | Datos | Estrategia |
|---|---|---|---|---|
| Lanzamiento | 1.000 | 30 | 1 GB | ECS min 2 tasks, RDS db.t3.medium |
| 6 meses | 10.000 | 100 | 10 GB | Mismo setup, monitorear metricas |
| 1 ano | 50.000 | 300 | 50 GB | ECS auto-scaling, RDS scale-up a db.r6g.large |
| 3 anos | 200.000 | 1.000 | 200 GB | Read replicas, particionamiento de tablas, evaluar microservicios |

## 3. Seguridad (STRIDE)

> Referencia: Microsoft STRIDE Threat Modeling

| Amenaza | Riesgo | Mitigacion |
|---|---|---|
| **Spoofing** (suplantacion de identidad) | Alto | JWT RS256 con keys rotadas cada 90d. KYC obligatorio con verificacion biometrica contra Core Bancario. MFA via SMS en operaciones financieras > $1M COP |
| **Tampering** (manipulacion de datos) | Alto | Audit trail inmutable (INSERT-only en PostgreSQL). HTTPS/TLS 1.3 en transito. Checksums en registros de audit. Firma digital en mensajes SQS |
| **Repudiation** (negacion de acciones) | Alto | Audit trail con timestamp, userId, IP, operacion, resultado para TODA operacion financiera. Retencion 10 anos (Ley 964/2005) |
| **Information Disclosure** (fuga de datos) | Critico | Row-Level Security en PostgreSQL (usuario solo accede a sus datos). Cifrado en reposo (RDS encrypted, S3 SSE). Cifrado en transito (TLS 1.3). No logging de datos sensibles (PII masking) |
| **Denial of Service** | Medio | WAF con rate limiting (100 req/min/usuario). API Gateway throttling. ECS auto-scaling con limite superior. CloudFront como CDN y DDoS protection |
| **Elevation of Privilege** | Alto | JWT con roles (investor, admin, system). Principio de menor privilegio en IAM. Service-to-service auth con JWT separado. Secrets Manager para credenciales |

## 5. Observabilidad (SA define QUE medir; SWA/Ops implementa)

**Plataforma elegida:** AWS CloudWatch + X-Ray (nativo, sin costo adicional de tooling)

### Metricas criticas que SA define

| Metrica | Umbral de alarma | Severidad | Por que importa |
|---|---|---|---|
| Disponibilidad API | < 99.5% en ventana 30 dias | P1 | SLO comprometido |
| Latencia P99 | > 500ms por 5 min consecutivos | P2 | QA-001 del PRD violado |
| Tasa de error 5xx | > 5% en ventana de 5 min | P1 | Posible incidente en curso |
| Queue depth SQS | > 100 mensajes por > 10 min | P2 | Backlog excesivo, Worker saturado |
| DLQ depth | > 0 mensajes por > 30 min | P2 | Operaciones financieras fallidas sin procesar |
| Inversiones E2E > 5 min | > 5% del total diario | P2 | QA-002 del PRD violado |
| Intentos de acceso no autorizado | > 10 en ventana de 1 hora | P1 | Posible ataque, QA-003 del PRD |
| Error budget consumido > 80% | -- | P2 | Congelar deploys no criticos |

### Lo que SWA/Ops implementa (fuera de scope SA)

- Formato de logs (JSON structured logging con campos estandar)
- Instrumentacion de X-Ray traces
- Dashboards en CloudWatch
- Configuracion de SNS para alertas
- Retencion y rotacion de logs

## 6. SPOFs (Single Points of Failure)

| SPOF | Impacto | Mitigacion actual | Mitigacion pendiente |
|---|---|---|---|
| Base de datos PostgreSQL | Perdida total de datos y operaciones | RDS Multi-AZ con failover automatico | Backups cross-region (evaluar costo) |
| Worker de Eventos (single instance) | Operaciones asincronas se detienen | ECS con min 2 tasks, SQS retiene mensajes hasta 14 dias | Health check con auto-replace |
| Audit Trail (tabla PostgreSQL) | No compliance — operaciones no se pueden ejecutar (fail-closed) | Multi-AZ, backups cada hora | Considerar DynamoDB append-only para audit (inmutabilidad nativa) |
| AWS Region (us-east-1) | Perdida total de servicio | N/A — single region en MVP | Multi-region en fase 2 si el negocio lo justifica |
| Deceval (unico proveedor custodia) | No se pueden registrar inversiones | DLQ con reintentos, operaciones quedan pendientes | N/A — Deceval es monopolio regulado |

## 7. Capacity Planning (SA define umbrales; Ops ejecuta)

### SA define:

| Recurso | Actual (MVP) | Proyeccion 6m | Trigger de escalado | Estrategia |
|---|---|---|---|---|
| ECS Tasks (API) | 2 | 2-4 | CPU > 70% por 5 min | Auto-scaling horizontal |
| ECS Tasks (Worker) | 2 | 2-4 | Queue depth > 50 por 5 min | Auto-scaling horizontal |
| RDS Instance | db.t3.medium (2 vCPU, 4GB) | db.r6g.large (2 vCPU, 16GB) | Memory > 80% o connections > 80% | Vertical scale-up |
| RDS Storage | 20 GB | 100 GB | Storage > 70% | Auto-expand habilitado |
| ElastiCache | cache.t3.micro | cache.t3.small | Memory > 70% | Vertical scale-up |
| S3 (reportes/docs) | < 1 GB | 10 GB | N/A — sin limite practico | N/A |

### Ops implementa:

- Configuracion de auto-scaling policies en ECS
- Alarmas de capacidad en CloudWatch
- Proceso de scale-up manual para RDS (requiere ventana de mantenimiento)
