# ADR-001: Patron Event-Driven con SQS para Integraciones Externas

## Estado

Aceptado

## Contexto y Problema

La plataforma de inversion necesita integrar con 3 sistemas externos que tienen
latencias variables y no controlamos:

| Sistema | Operacion | Latencia tipica | SLA |
|---|---|---|---|
| Deceval | Registro de custodia de CDT | 1-30 segundos | 99.5% |
| PSE/ACH | Transferencia de fondos | 2-15 segundos | 99.9% |
| Core bancario | Consulta KYC | 500ms-3 segundos | 99.9% |

Si el flujo de compra de CDT es sincrono end-to-end, el usuario esperaria entre
3.5 y 48 segundos — inaceptable para una experiencia digital. Ademas, si Deceval
tiene una caida temporal (su SLA permite 3.6 horas/mes de downtime), toda la
plataforma quedaria bloqueada.

Las operaciones financieras requieren idempotencia estricta: cada mensaje debe
procesarse exactamente una vez para evitar doble inversion o doble cobro.

## Decision Drivers

- **UX aceptable:** El usuario no debe esperar mas de 2 segundos para obtener confirmacion
  de que su operacion fue recibida
- **Resiliencia:** La plataforma debe seguir operando si un sistema externo tiene downtime temporal
- **Idempotencia:** Operaciones financieras no toleran duplicados
- **Experiencia del equipo:** El equipo conoce SQS y .NET BackgroundServices
- **Auditabilidad:** Toda operacion debe quedar en audit trail antes de ejecutarse

## Opciones Consideradas

- Opcion 1: REST sincrono end-to-end
- Opcion 2: Event-driven con SQS + REST para endpoints de lectura
- Opcion 3: Polling periodico (cliente consulta estado)

## Decision

**Opcion elegida: "Event-driven con SQS + REST para endpoints de lectura"**

### Justificacion

Porque desacopla la latencia de los sistemas externos del flujo del usuario,
permite reintentos automaticos con backoff exponencial, y soporta idempotencia
nativa via SQS deduplication.

A diferencia de REST sincrono (Opcion 1) que bloquea al usuario 3-48 segundos
y deja la plataforma vulnerable a downtime de terceros.

A diferencia de polling (Opcion 3) que agrega complejidad en el cliente y
genera carga innecesaria en la API.

## Pros y Contras de las Opciones

### Opcion 1: REST Sincrono End-to-End

- (+) Simple de implementar, el equipo lo conoce bien
- (+) Respuesta inmediata al usuario (exito o error)
- (-) Latencia de 3-48 segundos es inaceptable para UX
- (-) Si Deceval cae, toda la plataforma se bloquea
- (-) No hay retry automatico — el usuario debe reintentar manualmente
- (-) Riesgo de timeout y operaciones en estado indeterminado

### Opcion 2: Event-Driven con SQS — ELEGIDA

- (+) Usuario recibe confirmacion en < 2 segundos ("operacion en proceso")
- (+) Reintentos automaticos con backoff si Deceval/PSE fallan temporalmente
- (+) Dead Letter Queue (DLQ) para mensajes fallidos — no se pierde operacion
- (+) SQS FIFO con deduplicacion nativa — idempotencia incluida
- (+) El equipo conoce SQS y .NET BackgroundServices
- (-) Complejidad adicional: necesita Worker dedicado para procesar eventos
- (-) El usuario no ve resultado inmediato — necesita flujo de "estado de operacion"
- (-) Eventual consistency entre la API y el estado real de la operacion

### Opcion 3: Polling Periodico

- (+) Cliente simple que consulta estado cada N segundos
- (+) Sin dependencia de infraestructura de mensajeria
- (-) Carga innecesaria en la API (muchas consultas sin cambio de estado)
- (-) UX suboptima — el usuario no sabe cuando estara listo
- (-) No resuelve el problema de latencia del backend

## Consecuencias

### Positivas

- El usuario recibe confirmacion inmediata (< 2s) de que su operacion fue registrada
- La plataforma sigue operando si Deceval tiene downtime temporal (mensajes quedan en SQS)
- Operaciones fallidas van a DLQ — nunca se pierden, se pueden reprocesar
- Audit trail se escribe ANTES de enviar a procesamiento (fail-closed para compliance)

### Negativas / Trade-offs

- Se necesita un Worker separado para procesar mensajes de SQS — mas componentes a operar
- El usuario debe consultar el estado de su operacion (no es instantaneo)
- Eventual consistency: el portafolio puede tardar 1-30 segundos en reflejar la compra
- Si el DLQ acumula mensajes, se necesita proceso manual de reconciliacion

### Neutrales

- El patron event-driven es estandar en fintech — el equipo puede contratar talento que lo conozca
- SQS FIFO tiene limite de 300 msg/seg — suficiente para 300 inversiones/dia (CB-001)

## Validacion

- **Metrica:** Tiempo entre "usuario solicita compra" y "confirmacion de custodia en Deceval" < 5 minutos para el 95% de operaciones
- **Senal de alerta:** DLQ con mas de 10 mensajes acumulados por mas de 1 hora
- **Revision programada:** 3 meses post-lanzamiento — evaluar si la latencia promedio de Deceval justifica mantener el patron o simplificar
