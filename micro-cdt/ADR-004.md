# ADR-004: Modelo de KYC y Enrolamiento — API de Enrolamiento Digital Deceval

## Estado
Aceptado

---

## Contexto y Problema

Antes de que un usuario pueda invertir en un CDT, la plataforma debe verificar su identidad
(KYC — Know Your Customer) y vincularlo como participante en el sistema de valores colombiano.
Esta vinculación es un requisito regulatorio (Ley 964/2005 + normativa Deceval) y también el
primer punto de fricción con el usuario — si el proceso es lento o requiere presencia física,
se destruye la propuesta de valor de la plataforma.

El problema es: **¿qué mecanismo usa la plataforma para realizar el KYC y el enrolamiento
del usuario de forma 100% digital, con la validez legal requerida por la SFC?**

Fuerzas en juego:
- El KYC MUST tener validez legal bajo la normativa SFC y la Ley 1581/2012 (Habeas Data).
- La plataforma NO debe almacenar biometría ni documentos de identidad en sus propios servidores
  — esto aumenta el riesgo regulatorio de protección de datos.
- La firma del contrato de vinculación al mercado de valores requiere **firma electrónica
  certificada** bajo el Decreto 2364/2012.
- El objetivo de UX es: onboarding completo < 8 minutos desde el registro hasta la primera
  inversión disponible.
- Deceval ofrece un **servicio de enrolamiento digital** con firma electrónica certificada
  (OTP o biometría) para la vinculación de personas naturales al mercado de valores —
  confirmado en PQ-03 del RFC-001.
- El LOI firmado con el partner financiero incluye la integración con el canal de enrolamiento Deceval.

---

## Decision Drivers

- **DR-1 — Validez regulatoria:** El enrolamiento debe producir una vinculación reconocida por la SFC y Deceval. Solo la firma electrónica certificada cumple este requisito sin presencia física.
- **DR-2 — Experiencia digital completa:** La plataforma se diferencia de la banca tradicional por no requerir presencia física. Cualquier paso que requiera desplazamiento del usuario destruye la propuesta de valor.
- **DR-3 — Protección de datos:** La plataforma NO debe ser el custodio de biometría ni documentos de identidad — el riesgo regulatorio bajo Ley 1581/2012 es alto y la infraestructura de seguridad requerida supera la capacidad del piloto.
- **DR-4 — Time-to-market:** Construir un stack de KYC propio (OCR de documentos, liveness detection, motor de decisión) toma 4–6 meses. El MVP tiene 9 meses totales.
- **DR-5 — Calidad del enrolamiento:** La tasa de error del proceso de enrolamiento (usuarios que fallan y abandonan) impacta directamente la métrica de adquisición. Un proveedor especializado tiene tasas de éxito mejores que un KYC construido internamente.

---

## Opciones Consideradas

- **Opción A:** KYC propio (construir internamente: OCR documentos, liveness, motor de decisión)
- **Opción B:** KYC del core bancario del partner (delegar completamente al partner financiero)
- **Opción C:** API de enrolamiento digital Deceval con firma electrónica certificada (OTP/biometría) ← **ELEGIDA**

---

## Decisión

**Opción elegida: "API de enrolamiento digital Deceval"**

### Justificación

Porque Deceval es la autoridad de registro del mercado de valores colombiano, y su servicio de
enrolamiento digital produce directamente una vinculación con validez regulatoria SFC sin
requerir un paso adicional de homologación. La firma electrónica certificada (OTP o biometría)
cumple el Decreto 2364/2012, lo que hace que el contrato de vinculación sea legalmente válido
sin presencia física del usuario.

A diferencia de la Opción A (KYC propio), Deceval ya tiene el stack de verificación de identidad,
liveness detection y firma electrónica — construirlo internamente consumiría 4–6 meses y
exigiría que la plataforma almacene y custodie datos biométricos, aumentando la superficie
regulatoria bajo Ley 1581/2012.

A diferencia de la Opción B (KYC del partner bancario), la Opción C produce una vinculación
directa en Deceval — el usuario queda inscrito en el depósito de valores, no solo en el core
bancario del partner. Esto es necesario para que el CDT pueda custodiarse en Deceval a nombre
del usuario (ADR-001). Si se usa solo el KYC del partner, la vinculación Deceval requeriría
un paso adicional, duplicando el proceso.

**Flujo de enrolamiento elegido:**

```
Usuario en la plataforma
  → La plataforma llama API Deceval Enrolamiento (REST)
  → Deceval orquesta: captura documento + selfie/OTP + firma electrónica certificada
  → Deceval retorna: { deceval_user_id, estado: "ENROLADO", timestamp }
  → La plataforma almacena solo: deceval_user_id + estado + timestamp (NO biometría, NO documento)
  → El usuario queda habilitado para invertir
```

La plataforma actúa como orquestador de la UX — Deceval es el sistema de identidad.
La plataforma **NO almacena** ningún dato biométrico ni copia del documento de identidad.

---

## Pros y Contras de las Opciones

### Opción A: KYC Propio
- (+) Control total del flujo y la UX del onboarding
- (+) Sin dependencia de Deceval para el proceso de enrolamiento
- (+) Permite personalizar el proceso según el segmento de usuario
- (-) 4–6 meses de desarrollo solo para el stack de KYC — incompatible con el timeline del MVP
- (-) La plataforma se convierte en custodio de biometría y documentos de identidad — alta superficie bajo Ley 1581/2012
- (-) La firma electrónica propia requiere certificación adicional bajo Decreto 2364/2012
- (-) El resultado del KYC propio aún requiere una vinculación en Deceval — dos pasos separados

### Opción B: KYC del Core Bancario del Partner
- (+) El partner ya tiene el proceso de KYC operativo y certificado con la SFC
- (+) Reduce el número de integraciones — un solo proveedor para KYC + custodia
- (-) El KYC del partner vincula al usuario en el core bancario del partner, no directamente en Deceval
- (-) Requiere un paso adicional para la vinculación Deceval — el proceso de onboarding tiene dos fases separadas
- (-) La experiencia del usuario depende de los tiempos y la UX del partner bancario — puede requerir presencia física o procesos heredados del banco
- (-) Si se cambia de partner en Fase 2, todos los usuarios deben re-hacer el KYC
- (-) Acopla el usuario a la infraestructura de identidad del partner, no a Deceval directamente

### Opción C: API de Enrolamiento Digital Deceval ✅
- (+) La vinculación se produce directamente en Deceval — validez regulatoria SFC desde el primer momento
- (+) Firma electrónica certificada cumple Decreto 2364/2012 — sin presencia física
- (+) La plataforma no almacena biometría ni documentos — el custodio de identidad es Deceval
- (+) Compatible con el LOI firmado y el timeline del MVP (integración estimada en 3–4 semanas)
- (+) Si se cambia de partner en Fase 2, la vinculación Deceval del usuario persiste — no hay re-enrolamiento
- (-) La plataforma tiene dependencia de disponibilidad de la API de enrolamiento Deceval (SUPUESTO-07 del RFC-001)
- (-) Si Deceval modifica su API o su proceso de enrolamiento, la plataforma debe adaptarse
- (-) La tasa de error del enrolamiento (fallos de biometría, OTP no recibido) está parcialmente fuera del control del equipo

---

## Consecuencias

### Positivas
- El usuario queda registrado directamente en Deceval — la vinculación es portable (no atada al partner).
  Si en Fase 2 se cambia de partner, el `deceval_user_id` sigue siendo válido.
- La plataforma no es custodio de datos sensibles de identidad — reduce drásticamente la superficie
  de riesgo bajo Ley 1581/2012 y simplifica el DPA.
- El onboarding < 8 minutos es alcanzable porque Deceval tiene el proceso optimizado — no hay
  revisión manual en el flujo nominal.

### Negativas / Trade-offs
- La tasa de abandono en el onboarding depende parcialmente de la UX de Deceval (la parte de
  captura de documento/biometría). La plataforma debe medir esta métrica (sugerida en 🟢 S-01
  del RFC-001) para identificar si el abandono es en la parte propia o en el paso Deceval.
- Los fallos de enrolamiento (OTP no llega, biometría rechazada) deben tener un flujo de
  reintento claro en la UX. La Tech Spec DEBE definir los estados del proceso de enrolamiento
  y cómo se maneja cada tipo de error de la API Deceval.
- La plataforma solo almacena `deceval_user_id` + estado. Toda consulta sobre el estado del
  enrolamiento de un usuario debe ir a la API Deceval — no hay estado local que cachear de
  forma indefinida sin riesgo de inconsistencia.

### Neutrales
- El proceso de enrolamiento Deceval aplica tanto para OTP (usuario sin smartphone de alta
  gama) como para biometría (usuario con capacidad biométrica). La API maneja ambos flujos —
  la plataforma no necesita diferenciarlos en la lógica de negocio.

## Modelo de Datos del Usuario en la Plataforma

La plataforma almacena **únicamente** lo necesario para operar — no datos de identidad:

```
Tabla: usuarios
- id                UUID       PK
- email             VARCHAR    (credencial de acceso)
- telefono          VARCHAR    (para notificaciones, no para KYC)
- deceval_user_id   VARCHAR    (retornado por API Deceval al completar enrolamiento)
- estado_enrolamiento ENUM     (PENDIENTE | EN_PROCESO | ENROLADO | RECHAZADO)
- fecha_enrolamiento TIMESTAMPTZ
- fecha_registro    TIMESTAMPTZ

NO se almacena:
- Número de documento de identidad (custodio: Deceval)
- Biometría / selfie (custodio: Deceval)
- Copia del documento (custodio: Deceval)
```

## Validación

- **Métrica:** Tasa de conversión del enrolamiento (usuarios que inician y completan). Objetivo: ≥ 85% en piloto.
- **Métrica:** Tiempo promedio del proceso de enrolamiento end-to-end. Objetivo: < 8 minutos.
- **Métrica:** Tasa de error de la API Deceval enrolamiento. Objetivo: < 5%; alerta si > 5% en 1 hora.
- **Señal de alerta:** Si la tasa de abandono en el paso de Deceval supera el 20%, revisar si el problema es de UX de la plataforma (instrucciones, contexto) o de la API Deceval.
- **Revisión programada:** Q1 2027 — evaluar si el volumen de re-enrolamientos o errores justifica un mecanismo de fallback (ej. KYC asistido manual para casos excepcionales).
