| **Observabilidad** | AWS CloudWatch + X-Ray | — | Logs estructurados JSON, trazas distribuidas, alarmas SLA |

## 2. Estructura de Contenedores (C4 Level 2)

> Diagrama C4 L2 generado con Excalidraw MCP (pendiente — ver Fase 5 del orquestador).
> El texto a continuación es la especificación que guía la generación del diagrama.

### Contenedores de la Plataforma

| Contenedor | Tecnología | Responsabilidad | Tipo |
|---|---|---|---|
| **API Gateway** | AWS API Gateway HTTP API v2 | Punto de entrada único: throttling por ruta (`/auth`, `/inversiones`, `/portafolio`), validación JWT edge, acceso logging. Enruta al API de Inversiones vía VPC Link | Gateway |
| **Web App** | Next.js 14 / CloudFront + S3 | Interfaz de usuario: onboarding, compra, portafolio, autenticación. Web responsive (no nativa) | SPA/SSR |
| **API de Inversiones** | ASP.NET Core .NET 8 / ECS Fargate | Lógica de negocio: autenticación, enrolamiento, gestión de inversiones, portafolio, orquestación de integraciones síncronas | API REST |
| **Worker de Eventos** | .NET 8 Lambda (3 funciones con triggers separados) | Procesamiento asíncrono de eventos de negocio. **Tres triggers:** (1) EventBridge 22:00 COT → redención automática de CDTs; (2) EventBridge 02:00 COT → exportación audit trail a S3; (3) EventBridge 23:00 COT → reporte SFC via Core Bancario | Worker |
| **Base de Datos** | AWS RDS PostgreSQL 15 (Multi-AZ) | Persistencia principal: `usuarios`, `cdts`, `audit_events`, `sesiones`, `idempotency_keys`. Source of truth de todas las operaciones | Base de datos |
| **Cache** | AWS ElastiCache Redis Serverless | Sesiones JWT, catálogo CDTs del Core Bancario (TTL 5 min), rate limiting, idempotency keys en vuelo | Cache |
| **Audit Archivo** | AWS S3 Object Lock COMPLIANCE | Archivo regulatorio inmutable 10 años: eventos diarios comprimidos NDJSON + hash Merkle por día | Almacenamiento |
| **Cola SQS** | AWS SQS Standard (2 colas) | `cola-emision`: desacopla emisión CDT del path síncrono. `cola-redencion`: desacopla instrucciones batch del Worker de Eventos | Cola |

> **Nota sobre RDS vs Aurora:** Se elige RDS PostgreSQL 15 Multi-AZ sobre Aurora Serverless v2 porque el piloto requiere comportamiento predecible en disponibilidad (Multi-AZ estándar) y el equipo tiene más experiencia operativa con RDS. Aurora Serverless v2 puede retomarse en Fase 2 si el patrón de carga lo justifica.

### Sistemas Externos

| Sistema | Tipo | Rol en la solución |
|---|---|---|
| **Deceval — API Enrolamiento** | Externo regulado | KYC digital con firma electrónica certificada (ADR-004) — integración directa desde API de Inversiones |
| **Core Bancario del Partner** | Externo (LOI firmado) | Custodia y emisión de CDTs en Deceval (ADR-001); canal de reportes regulatorios hacia la SFC | 
| **PSE / Wompi** | Externo comercial | Débito y crédito de fondos del usuario (ADR-002) |
| **SFC (Superfinanciera)** | Ente regulatorio | Destinatario de reportes diarios — **no es una integración directa de la plataforma**. La plataforma genera y valida los datos; el Core Bancario los transmite a la SFC |
| **AWS SES** | Servicio AWS | Email transaccional: confirmaciones, alertas, recuperación |
| **AWS SNS / Twilio** | Servicio managed | SMS OTP para enrolamiento (CU-001) y 2FA (CU-005) |

> **Aclaración sobre SFC:** La plataforma no llama a ninguna API de la SFC directamente. El flujo es: Worker de Eventos genera el reporte → valida integridad → entrega al Core Bancario del Partner → el Core Bancario transmite a la SFC bajo su propia licencia. La responsabilidad de la plataforma es la **integridad y completitud de los datos** (RN-009 del PRD).

## 3. Máquina de Estados del CDT

> Resuelve 🟡 I-01 del PRD-001.

```
                    ┌─────────────────────────────────┐
                    │                                 │
[PENDIENTE_EMISION] ──── emisión exitosa ────► [ACTIVO]
         │                                       │
         │ error Deceval/partner                 │ Lambda 22:00 COT
         ▼                                       ▼
  [EMISION_FALLIDA]              [PENDIENTE_REDENCION]
  (→ devolución PSE)                     │
                                         │ partner confirma
                                         ▼
                              [REDENCION_EN_PROCESO]
                                         │
                               PSE crédito exitoso
                                         ▼
                                    [VENCIDO]
                                         │
                           ┌─────────────┴──────────────┐
                           │                             │
                  fallo PSE crédito            fallo partner timeout
                           ▼                             ▼
             [REDENCION_PENDIENTE_ACREDITACION]  [REDENCION_EN_PROCESO]
             (reintento < 4h, Ops alerta)        (manual Ops, SLA 2h hábiles)
```

**Transiciones válidas** (enforced en BD via CHECK constraint + código):
- Solo `ACTIVO → PENDIENTE_REDENCION` (por Lambda, no por API usuario)
- `EMISION_FALLIDA` es estado terminal que requiere intervención de Ops
- `VENCIDO` es estado terminal — no hay transición de salida

## 4. Contratos de Integración

### Contrato: API de Inversiones → API Enrolamiento Deceval

| Campo | Valor |
|---|---|
| **Protocolo** | REST/HTTPS |
| **Endpoint** | `POST /enrolamiento/persona-natural` (per documentación Deceval) |
| **Autenticación** | API Key en header `X-Deceval-Key` (rotación en Secrets Manager cada 90 días) |
| **Formato request** | JSON: `{ usuario_ref, callback_url }` — Deceval orquesta el resto |
| **Formato response** | JSON: `{ deceval_user_id, estado, timestamp }` |
| **Timeout configurado** | 30 segundos — si supera, estado `EN_PROCESO`, reintento por polling |
| **Idempotencia** | Deceval retorna el mismo `deceval_user_id` si el usuario ya existe |
| **SLA esperado** | Disponibilidad ≥ 99% (SUPUESTO-07 RFC-001) |
| **Fallback** | Estado `EN_PROCESO` + notificación por email + polling cada 5 min hasta 24h |
| **Estrategia de error** | 4xx → error de datos del usuario (mostrar al usuario). 5xx → retry 3x con backoff exponencial (1s, 4s, 16s) |

### Contrato: API de Inversiones → Core Bancario del Partner (Emisión CDT)

| Campo | Valor |
|---|---|
| **Protocolo** | REST/HTTPS + mTLS (seguridad adicional por naturaleza regulada) |
| **Endpoint** | `POST /cdt/emitir` |
| **Autenticación** | API Key + certificado mTLS (Secrets Manager) |
| **Formato request** | JSON: `{ deceval_user_id, monto_cop, plazo_dias, id_emisor, referencia_pse, idempotency_key }` |
| **Formato response** | JSON: `{ id_cdt_deceval, fecha_vencimiento, tasa_ea_efectiva, timestamp_emision }` |
| **Timeout configurado** | 30 segundos síncronos → si supera, estado `PENDIENTE_CONFIRMACION` + SQS retry |
| **Idempotencia** | `idempotency_key` = `referencia_pse` — el partner retorna el mismo CDT si ya fue emitido |
| **SLA esperado** | Respuesta < 30s en horario hábil (6am–8pm COT, días hábiles) |
| **Fallback** | `PENDIENTE_CONFIRMACION` → SQS → reintento cada 5 min × 6 → Ops Financieras alerta |
| **Estrategia de error** | 409 Conflict → CDT ya emitido (usar idempotency_key). 5xx → retry con backoff. Sin partner → modo solo lectura |

### Contrato: API de Inversiones → Wompi (Débito PSE)

| Campo | Valor |
|---|---|
| **Protocolo** | REST/HTTPS (redirección al flujo PSE de Wompi) |
| **Endpoint** | `POST /v1/payment_sources` + `POST /v1/transactions` |
| **Autenticación** | API Key pública (frontend) + API Key privada (backend) |
| **Flujo** | Backend crea transaction → Frontend redirige a Wompi PSE → Wompi notifica backend por webhook |
| **Webhook** | `POST /api/pagos/wompi-webhook` — verificado con signature HMAC |
| **Timeout** | El débito PSE puede tardar hasta 5 minutos (depende del banco del usuario) |
| **Idempotencia** | `reference` único por transacción = `{cdt_id}-{timestamp}` |
| **SLA esperado** | Tasa de éxito ≥ 97% (QA en PRD-001) |
| **Fallback** | Si Wompi no notifica en 10 min → consulta activa de estado → si DECLINED, no emitir CDT |

### Contrato: Worker de Eventos (trigger audit) → S3 Object Lock

| Campo | Valor |
|---|---|
| **Protocolo** | AWS SDK S3 PutObject |
| **Bucket** | `micro-cdt-audit-archive` — Object Lock COMPLIANCE, retención 10 años |
| **Key pattern** | `audit/{año}/{mes}/{día}/events.ndjson.gz` |
| **Contenido** | Todos los `audit_events` del día comprimidos en NDJSON + archivo `{día}.merkle.json` con hash raíz |
| **Trigger** | Lambda EventBridge cron: `0 2 * * *` (02:00 COT) |
| **Idempotencia** | Si el objeto ya existe para el día, el Lambda verifica el hash y sale sin reescribir |
| **Alerta** | CloudWatch Alarm si el Lambda no completa con éxito — escala a Ops en < 15 min |

### Contrato: Worker de Eventos (trigger redención) → Core Bancario del Partner

| Campo | Valor |
|---|---|
| **Protocolo** | REST/HTTPS + mTLS |
| **Endpoint** | `POST /cdt/redimir` |
| **Formato request** | JSON: `{ id_cdt_deceval, cuenta_pse_destino, idempotency_key }` |
| **Procesamiento** | Async — el partner procesa la redención con Deceval y notifica via webhook |
| **Idempotencia** | `idempotency_key` = `{id_cdt}-redencion` — seguro de reintentar |
| **SLA esperado** | Confirmación de redención < 2 horas hábiles |
| **Fallback** | Estado `REDENCION_EN_PROCESO` → Ops Financieras alerta con SLA manual de 2h hábiles |

## 5. Fitness Functions

> Automatismos que validan que la arquitectura sigue cumpliendo sus promesas.

| ID | Fitness Function | Tipo | Qué valida | Automatización | Frecuencia |
|---|---|---|---|---|---|
| **FF-001** | Audit trail sin mutaciones | Atómica | La tabla `audit_events` no tiene permisos UPDATE/DELETE para ningún rol de aplicación | Test en CI: conectar con rol de app → intentar UPDATE → debe fallar con error de permisos | En cada PR |
| **FF-002** | Aislamiento de datos entre usuarios | Atómica | Un token JWT de usuario A no puede acceder a datos del usuario B | Test E2E: crear 2 usuarios → autenticar A → intentar GET /portafolio/{usuario_B_id} → HTTP 403 | En cada PR |
| **FF-003** | Latencia de portafolio < 500ms p99 | Holística | CU-003 cumple QA-001 bajo carga real | Load test semanal (k6): 200 usuarios concurrentes, medir p99 | Semanal (CI nocturno) |
| **FF-004** | Exportación S3 nocturna sin gaps | Atómica | El audit archive no tiene días faltantes | Lambda de verificación: cuenta objetos S3 vs días transcurridos desde go-live; alerta si hay gap | Diaria (6am COT) |
| **FF-005** | Estado del CDT coherente con source of truth | Atómica | Ningún CDT tiene un estado en BD que contradiga el último evento en `audit_events` | Query de consistencia en BD: `cdts.estado` debe coincidir con el último evento registrado | Diaria (reconciliación) |
| **FF-006** | Integración hexagonal — no hay acoplamiento directo a partner | Atómica | El código de negocio no importa directamente el namespace del adapter del partner | Análisis estático en CI (ArchUnit .NET): dominio no debe referenciar `PartnerAdapter` namespace | En cada PR |
| **FF-007** | Sin secretos en código fuente | Atómica | No hay API keys, passwords ni certificados hardcodeados en el repo | Trufflehog/git-secrets en CI pre-commit + pre-push | En cada commit |
| **FF-008** | Reporte SFC diario a tiempo | Holística | El Worker de Reportes completa antes de las 23:30 COT | CloudWatch Alarm: si la Lambda no completa con SUCCESS antes de las 23:30 COT → PagerDuty Ops | Diaria |

## 6. Quality Attribute Trade-offs

| Trade-off | Decisión | Justificación |
|---|---|---|
| **Consistencia vs Latencia (escrituras)** | Consistencia primero | RN-003: el INSERT en `audit_events` DEBE completarse antes de confirmar una operación al usuario. Acepta latencia adicional en escrituras (~20-30ms) a cambio de cero pérdida de eventos financieros |
| **Latencia vs Acoplamiento (lecturas)** | Latencia primero | CU-003 (portafolio) se resuelve 100% desde la BD local — sin llamadas a partner ni Deceval en el path de lectura. El rendimiento se calcula localmente (QA-001 < 500ms p99) |
| **Seguridad vs Fricción de usuario** | Seguridad primero | 2FA obligatorio en cada login (CU-005). Acepta 1 paso adicional en la autenticación a cambio de QA-004 (0 accesos no autorizados). El público objetivo invierte dinero real — la fricción es aceptable |
| **Disponibilidad vs Coherencia (failover)** | Disponibilidad primero | Aurora Multi-AZ con RPO < 5 min. En failover, pueden existir operaciones `PENDIENTE_CONFIRMACION` que se reconcilian cuando el partner responde. Aceptamos coherencia eventual en < 24h para mantener disponibilidad ≥ 99.5% |
| **Acoplamiento vs Simplicidad (hexagonal)** | Acoplamiento controlado | La interfaz `ICustodiaPort` y `IPagosPort` agregan 2 capas de indirección. El equipo de 5 ingenieros acepta esta complejidad inicial porque ADR-001 y ADR-002 prevén cambio de partner/agregador en Fase 2 |
| **Procesamiento síncrono vs asíncrono** | Mixto | Débito PSE y consulta portafolio son síncronos (UX crítica). Emisión CDT y redención son async vía SQS (los sistemas externos no garantizan respuesta < 3s de forma consistente). El usuario ve estados intermedios claros |

## 7. Políticas a Nivel de Solución

> El SA define la política. El Software Architect / Tech Lead define la implementación.

| Política | Decisión SA | Nota para SWA |
|---|---|---|
| **Errores API** | Todos los errores siguen RFC 7807 Problem Details: `{ type, title, status, detail, instance }` | Middleware global en ASP.NET Core; correlationId en `instance` |
| **Versionamiento API** | URL-based: `/api/v1/` para el MVP. Versión mayor solo cuando hay breaking changes | Routing por prefijo; mantener v1 activa mínimo 6 meses tras lanzar v2 |
| **Autenticación** | JWT RS256, `access_token` 15 min, `refresh_token` 7 días rotativo, 2FA OTP SMS obligatorio en login. Tokens en httpOnly cookies (no localStorage) | ASP.NET Core JWT Bearer; refresh en endpoint dedicado; rotación de refresh token en BD |
| **Idempotencia en escrituras financieras** | Todas las operaciones de emisión y redención aceptan `Idempotency-Key` header (UUID v4). La plataforma persiste la key y retorna el mismo resultado si ya fue procesada | Storage de keys en Redis (TTL 24h) + tabla `idempotency_keys` en Aurora como fallback |
| **Audit trail transaccional** | El INSERT en `audit_events` es parte de la misma transacción BD que la operación de negocio. Si el INSERT falla, toda la transacción hace rollback (RN-003) | Transacción explícita en el repositorio; nunca fire-and-forget para audit events |
| **Secretos** | Todos los secretos (API keys, certificados, strings de conexión) en AWS Secrets Manager. Rotación automática cada 90 días. Cero secretos en variables de entorno de ECS | CDK configura `secretsmanager:GetSecretValue` en el task role; Lambda environment vacío |
| **Logging** | Logs JSON estructurados en CloudWatch. Campos obligatorios: `timestamp`, `level`, `correlationId`, `userId` (si autenticado), `service`, `message`. Sin PII en logs | Serilog con JSON formatter; correlationId inyectado desde middleware de request |
| **Rate limiting** | Endpoints de autenticación: 5 req/min por IP. Endpoints financieros: 10 req/min por usuario. Bloqueo temporal de IP con > 100 req/min | Redis sliding window counter; respuesta 429 con `Retry-After` header |
| **CORS** | Solo el dominio de la Web App tiene acceso. Sin `*` en ningún ambiente incluyendo desarrollo | ASP.NET Core CORS policy; lista blanca de origins en Secrets Manager |
| **Datos personales** | La plataforma NO almacena documento de identidad, biometría, ni foto del usuario (ADR-004). Solo `deceval_user_id` + estado + timestamps | Revisar cualquier PR que agregue campos de PII a tablas de usuarios con Compliance |

## 8. Modelo de Datos — Entidades Principales

> Nivel SA: entidades del dominio y sus relaciones. El SA no define tipos de columna ni índices — eso es responsabilidad del SWA.

```
usuarios
  id, email, telefono, deceval_user_id, estado_enrolamiento,
  fecha_enrolamiento, fecha_registro, estado_cuenta

cdts
  id, usuario_id (FK), id_cdt_deceval, monto_cop, plazo_dias, tasa_ea,
  fecha_inicio, fecha_vencimiento, estado, referencia_pse_debito,
  id_emisor, rendimiento_final

audit_events                              ← APPEND ONLY — sin UPDATE/DELETE
  id, timestamp, tipo_evento, usuario_id,
  entidad_id, payload (JSONB),
  hash_anterior (SHA-256), hash_actual (SHA-256)

sesiones                                  ← para refresh token rotation
  id, usuario_id (FK), refresh_token_hash,
  fecha_expiracion, revocada, ip_origen

idempotency_keys                          ← deduplicación de operaciones
  key (UUID), operacion, resultado_json, fecha_creacion
```

**Relaciones:**
- `usuarios` 1→N `cdts`
- `usuarios` 1→N `audit_events`
- `usuarios` 1→N `sesiones`
- `cdts` 1→N `audit_events` (via `entidad_id`)
