## 1. Diagrama de Arquitectura

**Diagrama C4 Level 2 (Container):** `micro-cdt/C4-L2-micro-cdt.excalidraw`

**Context Map DDD:** `micro-cdt/CM-001.md` — 5 bounded contexts:
- **Core:** Identidad y Enrolamiento, Inversiones
- **Supporting:** Pagos, Cumplimiento Regulatorio
- **Generic:** Catálogo de Productos

**Deployment lógico (AWS):**

| Contenedor | Hosting | Justificación |
|---|---|---|
| Web App (Next.js 14) | ECS Fargate | Sin gestión de servidores, escala a cero en piloto |
| API de Inversiones (.NET 8) | ECS Fargate | Mismo patrón, sin state |
| API Gateway | AWS HTTP API v2 | Managed, throttling nativo, integración VPC Link |
| Worker de Eventos | AWS Lambda .NET 8 | Tres triggers EventBridge — batch nocturno ideal para Lambda |
| Base de Datos | RDS PostgreSQL 15 Multi-AZ | HA + failover automático, `audit_events` append-only |
| Cache | ElastiCache Redis Serverless | Sin aprovisionamiento para piloto |
| Cola de Eventos | AWS SQS Standard | Decoupling API → Worker, at-least-once delivery |
| Audit Archivo | S3 Object Lock COMPLIANCE | Inmutabilidad regulatoria demostrable (ADR-003) |

## 2. Drivers de Arquitectura

### 2.1 Requisitos de Rendimiento

| Escenario | Latencia objetivo | Throughput piloto | Throughput 1 año | Medición |
|---|---|---|---|---|
| Consulta portafolio (CU-003) | < 500ms P99 | 20 RPS | 200 RPS | k6 semanal + CloudWatch P99 |
| Compra CDT — flujo completo (CU-002) | < 3 min end-to-end | 2 TPS | 20 TPS | Trace distribuido X-Ray |
| Login + 2FA (CU-005) | < 1s P99 | 5 RPS | 50 RPS | CloudWatch P99 |
| Worker redención batch (CU-004) | < 30 min total | 30 CDTs/noche | 500 CDTs/noche | Lambda duration metric |
| Exportación audit S3 (nightly) | < 15 min | 3.000 eventos/noche | 50.000 eventos/noche | Lambda duration + alerta |

> **Nota de scope SA:** Los objetivos de latencia son los SLOs que el SA garantiza al negocio.
> La instrumentación, dashboards y configuración de alarmas es responsabilidad de SWA/SRE.

### 2.2 Disponibilidad y Tolerancia a Fallos

| Componente | SLO plataforma | Estrategia HA | Failover |
|---|---|---|---|
| API de Inversiones (ECS) | 99.5% mensual | ≥ 2 tasks Fargate, AZ múltiples | ECS Service reemplaza task caída en < 30s |
| Base de Datos (RDS Multi-AZ) | 99.95% mensual | Standby en AZ diferente | Failover automático RDS < 60s |
| Cache (Redis Serverless) | 99.9% mensual | Serverless managed por AWS | Cache miss → hit directo a RDS (no falla negocio) |
| Cola SQS | 99.99% (AWS SLA) | Managed, replicación interna AWS | N/A — AWS garantiza |
| Worker de Eventos (Lambda) | 99.9% mensual | Managed, reintentos automáticos | SQS DLQ captura fallos; re-proceso manual |
| S3 Object Lock | 99.99% (AWS SLA) | Managed, replicación interna AWS | N/A |

**SLO global de la plataforma:** 99.5% mensual de disponibilidad del flujo crítico
(compra + portafolio). Degraded mode: portafolio disponible aunque Deceval/Core Bancario no estén.

**RPO (Recovery Point Objective):** 24 horas.
- Justificación: RDS Multi-AZ tiene réplica síncrona — pérdida máxima de datos en failover es < 1 min.
  El RPO de 24h aplica al escenario de corrupción de datos donde se requiere restore de backup.
- Los backups automáticos de RDS se retienen 7 días.

**RTO (Recovery Time Objective):** 4 horas.
- Justificación: En un escenario de pérdida total (región AWS), el equipo de SRE necesita
  4 horas para levantar el stack en otra región usando IaC (CDK). Para el piloto, una sola
  región (us-east-1) es aceptable — la SFC no exige DR cross-region en sandbox.
- Para producción plena (post-sandbox): RTO < 1 hora con región secundaria activa.

### 2.3 Escalabilidad

| Horizonte | Usuarios activos | Escrituras/día | Lecturas/día | Estrategia |
|---|---|---|---|---|
| Piloto (Q4 2026) | < 1.000 | 100 | 2.000 | 2 tasks Fargate mínimo, Redis t3.micro |
| 6 meses (Q2 2027) | 10.000 | 500 | 8.000 | Auto-scaling ECS (CPU > 70%), Redis r7g.large |
| 1 año (Q4 2027) | 50.000 | 3.000 | 20.000 | ECS 10 tasks max, RDS db.r6g.large, Redis cluster |
| 3 años | 200.000 | 10.000 | 80.000 | 🔴 TODO: evaluar si RDS sigue siendo suficiente o requiere sharding/read replicas |

**Cuello de botella anticipado:** La Base de Datos es el primer cuello de botella. A 10.000
escrituras/día, Aurora PostgreSQL 15 con 1 primaria es suficiente. A 50.000+/día se debe
evaluar una read replica para las consultas de portafolio (CU-003).

## 3. Seguridad — Modelo de Amenazas STRIDE

> Referencia: Microsoft STRIDE. Scope: flujo de compra de CDT y autenticación.
> El SA define el modelo de amenazas. El SWA implementa las mitigaciones.

| Amenaza STRIDE | Vector | Riesgo | Mitigación implementada | Estado |
|---|---|---|---|---|
| **Spoofing** (identidad falsa) | Login con credenciales robadas | 🔴 Alto | JWT RS256 (15 min) + OTP SMS 2FA + lockout 5 intentos/15 min (CU-005) | Mitigado |
| **Spoofing** (impersonación webhook) | Webhook Wompi falsificado | 🔴 Alto | HMAC-SHA256 signature validation en header `x-wompi-signature` (TS-001) | Mitigado |
| **Tampering** (alteración de audit trail) | DBA borra/modifica `audit_events` | 🔴 Crítico | Append-only RDS (IAM role sin UPDATE/DELETE) + S3 Object Lock COMPLIANCE + hash encadenado SHA-256 (ADR-003) | Mitigado |
| **Tampering** (alteración en tránsito) | MITM entre servicios | 🔴 Alto | TLS 1.2+ en todas las conexiones; mTLS hacia Core Bancario (ADR-001) | Mitigado |
| **Repudiation** (negación de operaciones) | "Yo no compré ese CDT" | 🔴 Alto | Audit trail inmutable con `usuario_id`, `timestamp`, `payload_json`, `hash_anterior` (ADR-003) | Mitigado |
| **Information Disclosure** (fuga de datos biométricos) | Breach de BD interna | 🔴 Crítico | La plataforma NO almacena biometría ni documentos (ADR-004). Solo `deceval_user_id` + estado | Mitigado por diseño |
| **Information Disclosure** (secrets en código) | Secrets hardcodeados en repo | 🔴 Alto | AWS Secrets Manager, rotación cada 90 días, Trufflehog en CI (FF-007) | Mitigado |
| **Information Disclosure** (datos financieros en logs) | Números de cuenta o monto en logs | 🟡 Medio | SA Policy: logs estructurados JSON con `correlationId` — sin PII ni montos en campos de texto libre | Pendiente validación |
| **Denial of Service** (flood de endpoints) | Brute force en `/auth` | 🟡 Medio | API Gateway throttling: `/auth` 5 req/min/IP, `/inversiones` 10 req/min/user (TS-001) | Mitigado |
| **Denial of Service** (abuso de compras) | Múltiples compras simultáneas | 🟡 Medio | Idempotency-key en Redis TTL 24h + fallback Aurora — operación duplicada rechazada (TS-001) | Mitigado |
| **Elevation of Privilege** (acceso a audit_events) | App role con permisos excesivos | 🔴 Alto | IAM role de la aplicación: solo INSERT en `audit_events`, sin UPDATE/DELETE en ninguna tabla de dominio (FF-001) | Mitigado |
| **Elevation of Privilege** (acceso a Secrets Manager) | Lambda con permisos excesivos | 🟡 Medio | IAM least privilege por componente: cada ECS task y Lambda tiene su propio role con permisos mínimos | Pendiente auditoría |

### Amenazas fuera de scope (Separate Ways — gestionadas por terceros)

| Amenaza | Quien la gestiona |
|---|---|
| Seguridad de la API Deceval | Deceval bajo regulación SFC |
| Seguridad del Core Bancario | Partner financiero bajo licencia SFC |
| Seguridad de PSE/Wompi | Wompi bajo certificación PCI-DSS |
| Fraude en transacciones PSE | Wompi + banco del usuario |

## 4. Dependencias Externas

| Sistema | SLA contratado | Timeout configurado | Fallback si no responde |
|---|---|---|---|
| **Deceval — Enrollment API** | ≥ 99% (SUPUESTO-07 — sin SLA contractual firmado aún) | 30s + polling fallback hasta 10 min (TS-001) | Enrolamiento queda en estado `EN_PROCESO`; usuario puede retomar |
| **Core Bancario del Partner** | 🔴 Sin SLA contractual definido (TS-TODO-02) | 15s por llamada síncrona | Emisión/redención falla → CDT queda en `EMISION_FALLIDA` / `REDENCION_PENDIENTE_ACREDITACION`; reintentos vía SQS |
| **PSE / Wompi** | ~99.5% (SLA público Wompi) | 10 min ventana de confirmación de webhook | Si webhook no llega en 10 min → consulta activa al API Wompi; si sigue sin respuesta → CDT en `PENDIENTE_EMISION`, alerta a Ops |
| **AWS SQS** | 99.99% (AWS SLA) | N/A — async | N/A |
| **AWS S3 Object Lock** | 99.99% (AWS SLA) | N/A — async Lambda | Alerta si Lambda de exportación falla; re-proceso manual el siguiente día |
| **AWS RDS PostgreSQL** | 99.95% Multi-AZ (AWS SLA) | 5s para queries operacionales | Redis cache para lecturas de portafolio; failover automático < 60s |

### Riesgo crítico de dependencias

> 🔴 **Riesgo DR-EXT-01:** El Core Bancario del Partner no tiene SLA contractual documentado.
> El equipo DEBE obtener el SLA del partner antes de salir a producción (Q3 2026).
> Sin SLA, el SLO de 99.5% de la plataforma no es garantizable end-to-end.

## 5. Observabilidad

> **Plataforma:** AWS CloudWatch + X-Ray (decisión en TS-001, sin ADR separado).
> **Alertas críticas:** PagerDuty para SFC report y anomalías de audit trail.
> El SA define QUÉ medir y los umbrales. SWA/SRE implementa dashboards y alertas.

### 5.1 Métricas críticas (SA define umbrales)

| Métrica | Umbral de alarma | Severidad | Por qué importa |
|---|---|---|---|
| Disponibilidad del flujo compra CDT | < 99.5% en ventana 30 días | 🔴 P1 | SLO comprometido — impacta negocio y regulación |
| Latencia P99 `GET /portafolio` | > 500ms en ventana 5 min | 🟡 P2 | QA-001 — degradación de experiencia |
| Latencia P99 flujo compra end-to-end | > 3 min en cualquier transacción | 🔴 P1 | QA-002 — falla funcional del producto |
| Tasa de error HTTP 5xx | > 1% en ventana 5 min | 🔴 P1 | Posible incidente en producción |
| Tasa de error HTTP 4xx en `/auth` | > 20% en ventana 1 min | 🟡 P2 | Posible ataque de brute force |
| SQS queue depth | > 500 mensajes durante 15 min | 🟡 P2 | Worker de Eventos atrasado |
| Lambda Worker — duración total nightly | > 30 min | 🟡 P2 | Riesgo de que reporte SFC no llegue antes de las 23:00 COT |
| S3 export gap | > 24h sin objeto nuevo en bucket de audit | 🔴 P1 | Brecha en audit trail regulatorio |
| CDTs en estado `EMISION_FALLIDA` | > 0 CDTs en este estado por > 1h sin resolución | 🟡 P2 | Dinero debitado sin CDT emitido — requiere acción de Ops |
| CDTs en estado `REDENCION_PENDIENTE_ACREDITACION` | > 0 por > 24h | 🔴 P1 | Vencimiento sin pago al usuario |
| Tasa de error API Deceval enrollment | > 5% en ventana 1h | 🟡 P2 | Abandono en onboarding (ADR-004 — objetivo < 5%) |
| Reporte SFC entregado a Core Bancario | No entregado antes de 23:30 COT | 🔴 P1 | Riesgo regulatorio — reportar a Compliance inmediatamente |

### 5.2 Eventos de dominio a loguear (obligatorio)

Todos los eventos de dominio listados en CM-001 deben producir una entrada en `audit_events`:

| Evento | Campos mínimos requeridos en log |
|---|---|
| `UsuarioEnrolado` | `deceval_user_id`, `usuario_id`, `timestamp`, `correlationId` |
| `CDTEmitido` | `id_cdt_deceval`, `usuario_id`, `monto`, `tasa`, `referencia_pse`, `correlationId` |
| `EmisionFallida` | `referencia_pse`, `motivo`, `usuario_id`, `correlationId` |
| `CDTPendienteRedencion` | `id_cdt_deceval`, `usuario_id`, `monto_principal`, `rendimiento` |
| `RedencionCompletada` | `id_cdt_deceval`, `usuario_id`, `monto_total`, `referencia_credito_pse` |
| `AuditTrailExportado` | `fecha`, `ruta_s3`, `hash_merkle`, `total_eventos` |

> **SA Policy (TS-001):** NINGÚN log debe contener PII (número de documento, biometría, datos bancarios).
> Solo `deceval_user_id` como identificador cruzado. El SWA implementa el structured logging
> con esta restricción como contrato de arquitectura.

### 5.3 Trazas distribuidas

X-Ray debe instrumentar los siguientes flujos como trazas end-to-end:
1. Compra CDT: `POST /inversiones/cdts` → Deceval → Core Bancario → Wompi → SQS → Worker
2. Onboarding: `POST /auth/enrolamiento` → Deceval API
3. Worker nightly: EventBridge → Lambda → S3 + Core Bancario

El `correlationId` se propaga en headers HTTP y como campo en todos los logs y eventos.

## 6. SPOFs (Single Points of Failure)

| SPOF | Impacto si falla | Mitigación actual | Mitigación pendiente | Prioridad |
|---|---|---|---|---|
| **RDS PostgreSQL primaria** | Toda la plataforma offline | Multi-AZ: failover automático < 60s | Ninguna adicional para piloto | ✅ Mitigado |
| **API Gateway (AWS)** | Toda la plataforma offline | AWS managed, SLA 99.99% | Ninguna — confiamos en AWS | ✅ Mitigado |
| **ECS Fargate — API de Inversiones** | Toda la plataforma offline | ≥ 2 tasks en AZs diferentes | Auto-scaling configurado | ✅ Mitigado |
| **Deceval — Enrollment API** | Onboarding bloqueado (usuarios nuevos no pueden invertir) | Estado `EN_PROCESO` + polling; usuarios existentes no afectados | Circuit breaker + alerta Ops | 🟡 Parcial |
| **Core Bancario del Partner** | Emisión y redención de CDTs bloqueada | SQS + reintentos automáticos; `EMISION_FALLIDA` con alerta | ADR-001: Phase 2 directo a Deceval; hasta entonces, SPOF aceptado | 🔴 SPOF crítico |
| **PSE / Wompi** | Pagos bloqueados (compras y redencioness imposibles) | Consulta activa si webhook no llega en 10 min | ADR-002: migrar a PSE directo si > 10K tx/mes; hasta entonces, SPOF aceptado | 🟡 SPOF aceptado |
| **Worker de Eventos (Lambda nightly)** | Reporte SFC no entregado + audit trail no archivado | SQS DLQ + alerta PagerDuty P1 | Procedimiento manual en runbook (RB-001) para re-proceso | 🟡 Parcial |
| **Secrets Manager** | Aplicación no puede arrancar (no accede a credenciales) | AWS managed, SLA 99.9% | Cache local en memoria al arranque (solo lectura) | 🟡 Parcial |

### Resumen de riesgo residual

El **Core Bancario del Partner** es el SPOF más crítico por ser un único punto sin
redundancia alternativa en MVP. Es inherente al modelo de negocio del piloto (ADR-001).
El riesgo se acepta explícitamente hasta que el volumen justifique la integración directa
con Deceval en Fase 2 (Q2 2027+).

| Horizonte | Trigger de escalado | Acción | Recurso clave |
|---|---|---|---|
| Q2 2027 (10K usuarios) | CPU ECS > 70% durante 5 min | Auto-scaling: +2 tasks ECS | ECS Fargate |

### 7.3 Señal de capacidad agotada

Si se superan los siguientes umbrales, SRE debe escalar inmediatamente y notificar al SA:
- RDS CPU > 85% durante 15 min sostenidos
- ECS CPU > 80% en todas las tasks durante 10 min
- Redis evictions > 0 (significa que la caché está expulsando datos activos)
- SQS queue depth > 1.000 mensajes durante 30 min

## Crítica Automática (Fase 4)

### 🔴 Críticos

**🔴 C-01: El Core Bancario del Partner no tiene SLA contractual documentado.**
Es el SPOF más crítico del sistema y el que tiene mayor probabilidad de fallar en un piloto
(integraciones bancarias históricamente inestables). Sin un SLA firmado antes de Q3 2026,
la plataforma no puede comprometer un SLO de 99.5% al negocio. **Acción:** el equipo comercial
DEBE conseguir un SLA del partner antes de salir a producción o el SLO debe ajustarse a reflejar
la incertidumbre.

**🔴 C-02: RTO de 4 horas es teórico sin un ejercicio de DR documentado.**
El RTO de 4h asume que el equipo puede levantar el stack en otra región usando CDK. Sin un
runbook de DR ejecutable ni un ejercicio de simulación realizado, este RTO es aspiracional.
**Acción:** el RB-001 (Runbook) DEBE incluir el procedimiento de DR step-by-step. Se recomienda
un simulacro de DR antes del lanzamiento al piloto.

### 🟡 Importantes

**🟡 I-01: El modelo de STRIDE no cubre el flujo de redención automática (Worker nightly).**
El Worker de Eventos tiene acceso a la Base de Datos y al Core Bancario. Si el Worker es
comprometido (ej. vulnerabilidad en la dependencia Lambda), podría ejecutar redencioness no
autorizadas. El análisis STRIDE actual es suficiente para el MVP, pero el equipo de seguridad
debería revisar el Worker como surface de ataque en Q1 2027.

**🟡 I-02: La métrica de latencia end-to-end de la compra CDT (< 3 min) depende de Wompi.**
La latencia del flujo de compra depende del webhook de Wompi (confirmación PSE). Si Wompi
tarda > 2 min en notificar el débito, el SLO de 3 min se viola — pero no es una falla de la
plataforma. El sistema DEBE distinguir entre SLO de la plataforma y tiempo de Wompi en los
dashboards para evitar falsos positivos en las alertas.

**🟡 I-03: El Capacity Planning a 3 años es incompleto.**
La partición de `audit_events` en PostgreSQL se marca como TODO. A 200K usuarios y ~10K
transacciones/día durante 3 años, la tabla tendrá ~10M filas — manejable sin particionamiento,
pero la consulta de rangos históricos empezará a degradarse. Definir la estrategia de
particionamiento (por fecha) antes de que la tabla supere los 5M registros (~18 meses).

### 🟢 Sugerencias

**🟢 S-01:** Agregar una métrica de negocio (no solo técnica) al dashboard de observabilidad:
"CDTs emitidos en las últimas 24h" y "CDTs redimidos en las últimas 24h". Estas métricas
permiten detectar anomalías de negocio (ej. caída del Core Bancario) antes de que aparezcan
en las métricas técnicas.

**🟢 S-02:** Documentar el presupuesto de error mensual explícito para el SLO de 99.5%:
`100% - 99.5% = 0.5%` → **~3.6 horas de downtime permitido por mes**. Compartir este número
con el Product Owner para alinear expectativas de mantenimiento y ventanas de despliegue.

**🟢 S-03:** Definir un SLI (Service Level Indicator) separado para el flujo de onboarding
vs. el flujo de inversión. Un usuario que ya completó KYC no debería ver degradación si la
API de Deceval está caída. El dashborad debería mostrar la disponibilidad de cada flujo por
separado.
