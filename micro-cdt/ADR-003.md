# ADR-003: Arquitectura de Audit Trail Inmutable — Append-Only RDS + S3 Object Lock

## Estado
Aceptado

---

## Contexto y Problema

La regulación SFC exige que todas las operaciones financieras de la plataforma (compras, redencion
de CDTs, movimientos de fondos, cambios de estado de instrumentos) queden registradas en un
**audit trail inmutable con retención mínima de 10 años**. Inmutable significa que ningún
registro puede modificarse ni eliminarse una vez escrito — ni siquiera por el administrador
de base de datos.

El problema es: **¿qué arquitectura técnica garantiza la inmutabilidad del audit trail bajo
el stack AWS + .NET del equipo, con costo y complejidad apropiados para un piloto?**

Fuerzas en juego:
- La inmutabilidad MUST ser demostrable ante la SFC — no es solo una propiedad de código,
  sino una garantía del almacenamiento subyacente.
- El audit trail debe ser consultable en tiempo real para operaciones recientes (< 2 años)
  y archivable para el histórico regulatorio (2–10 años).
- El equipo usa AWS — aprovechar servicios managed reduce la carga operativa.
- Los datos del audit trail son append-only por definición: no hay UPDATE ni DELETE válido.
- La plataforma es el **source of truth** de todas las operaciones (RFC-001, Sección 1).
- Escala del piloto: ~3,000 transacciones escritura/día → ~1M registros/año → ~10M en 10 años.
  Volumen manejable, no requiere arquitectura de big data.

---

## Decision Drivers

- **DR-1 — Inmutabilidad demostrable:** La SFC puede auditar. La solución debe poder demostrar que ningún registro fue alterado, incluso con acceso de administrador a la infraestructura.
- **DR-2 — Consultabilidad operativa:** Ops Financieras y Compliance necesitan consultar el audit trail en tiempo real para el reporte diario SFC. La solución debe responder queries eficientemente sobre datos recientes.
- **DR-3 — Retención 10 años a bajo costo:** El costo de almacenamiento a largo plazo debe ser despreciable. Los datos del piloto crecerán gradualmente.
- **DR-4 — Simplicidad operativa:** El equipo de 5 ingenieros no tiene capacidad para operar infraestructura de almacenamiento compleja. AWS managed services obligatorio.
- **DR-5 — Verificabilidad de integridad:** La solución debe permitir verificar en cualquier momento que el conjunto de registros no ha sido alterado desde su escritura.

---

## Opciones Consideradas

- **Opción A:** Append-only table en RDS PostgreSQL con políticas DDL restrictivas
- **Opción B:** Amazon S3 Object Lock (WORM) como almacenamiento primario con DynamoDB para índices
- **Opción C:** Append-Only RDS PostgreSQL (operacional) + S3 Object Lock (archivo) con hashing encadenado ← **ELEGIDA**

---

## Decisión

**Opción elegida: "Append-Only RDS PostgreSQL + S3 Object Lock con hash encadenado"**

### Justificación

Porque la consultabilidad operativa (reportes SFC diarios, queries de Ops) requiere SQL con
índices eficientes, y S3 solo no satisface ese requisito. Sin embargo, RDS solo no satisface
la inmutabilidad demostrable ante un auditor — un DBA con permisos de superusuario puede
alterar datos en PostgreSQL a pesar de las restricciones a nivel de aplicación.

La arquitectura elegida combina dos capas:

**Capa 1 — Operacional (RDS Aurora PostgreSQL):**
- Tabla `audit_events` con columnas append-only: `id`, `timestamp`, `tipo_evento`, `payload_json`,
  `hash_anterior`, `hash_actual` (SHA-256 del payload + hash_anterior — encadenamiento tipo blockchain-lite).
- Sin triggers de DELETE ni UPDATE en esa tabla (enforced por IAM + políticas RDS).
- Accesible para queries en tiempo real. Retención: 2 años en RDS (costo operativo justificado).

**Capa 2 — Archivo regulatorio (S3 Object Lock):**
- Cada noche, un proceso Lambda exporta los eventos del día a S3 en formato NDJSON comprimido.
- El bucket tiene **S3 Object Lock en modo COMPLIANCE** con retención de 10 años — ni AWS
  Support puede borrar estos objetos durante el período de retención.
- Cada archivo exportado incluye el hash raíz del día (Merkle root) para verificación de integridad.
- Costo estimado: ~$15–30 USD/mes para el volumen del piloto.

**Verificación de integridad:** En cualquier momento, un auditor puede recalcular el hash
encadenado de cualquier rango de eventos y verificar que coincide con el hash raíz exportado
a S3 — demostrando que los datos no han sido alterados desde la exportación.

A diferencia de la Opción A (solo RDS), esta arquitectura satisface la inmutabilidad demostrable
ante la SFC porque S3 Object Lock COMPLIANCE es una garantía del almacenamiento, no del código.

A diferencia de la Opción B (S3 como primario), satisface la consultabilidad operativa sin
requerir Athena/Glue para cada query de Ops Financieras.

---

## Pros y Contras de las Opciones

### Opción A: Append-Only RDS PostgreSQL (solo)
- (+) Un solo sistema — simple de operar y de entender
- (+) SQL nativo — queries eficientes para cualquier tipo de reporte
- (+) El equipo ya conoce PostgreSQL
- (-) La inmutabilidad es a nivel de aplicación — un DBA con superusuario puede alterar datos
- (-) No satisface "inmutabilidad demostrable" ante auditor externo (SFC)
- (-) El costo de retención 10 años en RDS es significativo (~$200–400/mes a largo plazo)

### Opción B: S3 Object Lock como Almacenamiento Primario + DynamoDB para Índices
- (+) S3 Object Lock COMPLIANCE es inmutabilidad garantizada por AWS — demostrable ante SFC
- (+) Costo de almacenamiento mínimo a largo plazo
- (-) Queries operacionales (reportes diarios SFC) requieren Athena o Glue — latencia de minutos, no segundos
- (-) DynamoDB como índice complica el modelo de datos y requiere gestión adicional
- (-) La complejidad operativa supera la capacidad del equipo de 5 ingenieros

### Opción C: Append-Only RDS + S3 Object Lock + Hash Encadenado ✅
- (+) RDS satisface la consultabilidad operativa con SQL eficiente
- (+) S3 Object Lock COMPLIANCE satisface la inmutabilidad demostrable ante SFC
- (+) Hash encadenado permite verificación de integridad de extremo a extremo
- (+) Costo operativo bajo: RDS cubre 2 años recientes, S3 cubre los 10 años regulatorios
- (+) Proceso Lambda de exportación nocturna es simple de implementar y operar en AWS
- (-) Dos sistemas a operar (RDS + S3) — más superficie que la Opción A
- (-) El proceso de exportación nocturna debe ser monitoreado — si falla, el archivo regulatorio tiene un gap
- (-) La reconciliación entre RDS y S3 agrega un paso de verificación en las auditorías

## Consecuencias

### Positivas
- La SFC puede auditar el sistema en cualquier momento y verificar que los registros en S3 Object
  Lock no han sido modificados — inmutabilidad garantizada por el almacenamiento, no por políticas
  de código que pueden ser bypasseadas.
- Los reportes diarios SFC se generan desde RDS con queries SQL estándar — Ops Financieras puede
  operar sin conocimiento de la capa S3.
- El costo del piloto es despreciable: ~$15–30 USD/mes en S3 + ~$50–80 USD/mes en RDS Aurora Serverless.

### Negativas / Trade-offs
- El proceso Lambda de exportación nocturna es un punto de fallo que DEBE tener monitoreo y
  alertas. Si falla silenciosamente durante varios días, hay un gap en el archivo regulatorio.
  La Tech Spec MUST definir el mecanismo de alerta y el procedimiento de re-exportación.
- La ventana de inmutabilidad fuerte es nocturna — los eventos del día actual están solo en RDS
  (sin la garantía S3) hasta la exportación. Riesgo mitigado: la probabilidad de un incidente
  de integridad en las últimas horas del día es baja; la SFC no audita en tiempo real.

### Neutrales
- El hash encadenado (SHA-256 sobre payload + hash anterior) es suficiente para la verificación
  de integridad — no requiere blockchain ni tecnología distribuida. El concepto es equivalente
  al de una cadena de hash de un árbol Merkle simplificado.
- La tabla `audit_events` en RDS tendrá un volumen de ~1M filas/año. A 10 años en RDS, serían
  ~10M filas — perfectamente manejable en Aurora PostgreSQL con índices apropiados sobre
  `timestamp` y `tipo_evento`.

## Diseño de la Tabla `audit_events`

```sql
CREATE TABLE audit_events (
    id            UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    timestamp     TIMESTAMPTZ NOT NULL DEFAULT now(),
    tipo_evento   VARCHAR(64) NOT NULL,   -- ej: CDT_EMITIDO, PAGO_DEBITADO, REDENCION_PROCESADA
    usuario_id    UUID,
    entidad_id    UUID,                   -- ID del CDT, transacción PSE, etc.
    payload       JSONB       NOT NULL,   -- datos completos del evento
    hash_anterior CHAR(64)    NOT NULL,   -- SHA-256 del registro anterior (0x00...00 para el primero)
    hash_actual   CHAR(64)    NOT NULL    -- SHA-256(payload || hash_anterior)
);

-- Solo INSERT permitido. Enforced por IAM role de la aplicación (sin UPDATE/DELETE).
-- Índices operacionales:
CREATE INDEX idx_audit_timestamp ON audit_events(timestamp DESC);
CREATE INDEX idx_audit_usuario    ON audit_events(usuario_id, timestamp DESC);
CREATE INDEX idx_audit_entidad    ON audit_events(entidad_id, timestamp DESC);
CREATE INDEX idx_audit_tipo       ON audit_events(tipo_evento, timestamp DESC);
```

## Validación

- **Métrica:** Tasa de éxito del proceso Lambda de exportación nocturna. Objetivo: 100%. Alerta si < 100% en cualquier noche.
- **Métrica:** Tamaño del gap entre último evento en RDS y último evento en S3. Objetivo: < 24 horas.
- **Verificación de integridad:** Ejecutar recálculo completo de hash encadenado trimestralmente y comparar con hashes raíz en S3.
- **Señal de alerta:** Cualquier discrepancia en el hash encadenado → incidente Crítico inmediato.
- **Revisión regulatoria:** En la primera auditoría SFC (estimada Q2 2027), presentar el mecanismo de verificación de integridad al auditor.
